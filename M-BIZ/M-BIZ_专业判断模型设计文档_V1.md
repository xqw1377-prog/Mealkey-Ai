## 1. 文档概述

### 1.1 文档目的

本文档旨在定义 M-BIZ 专业判断模型（Professional Judgment Model, PJM）的整体架构、核心组件、交互流程与技术规范，为后续开发、测试与部署提供统一的设计参考。

### 1.2 适用范围

- 目标用户：企业决策者、业务分析师、领域专家
- 应用场景：商业判断、风险评估、策略推荐、合规审查
- 模型版本：V1（初始版本）

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **专业判断** | 基于领域知识、经验规则与数据证据的综合推理结论 |
| **推理引擎** | 核心逻辑处理模块，负责规则匹配与链式推理 |
| **置信度** | 模型对判断结果的确信程度，取值范围 [0, 1] |
| **上下文** | 影响判断的环境变量、历史记录与约束条件 |

## 2. 核心概念与定位

### 2.1 什么是专业判断模型

M-BIZ 专业判断模型模拟人类专家在面对复杂商业问题时的思考过程，将领域知识、经验法则与数据驱动分析相结合，输出结构化的判断结论与决策建议。

### 2.2 模型能力边界（V1）

| 能力维度 | 范围 | 说明 |
|----------|------|------|
| 领域覆盖 | 通用商业判断 | 后续版本可扩展至垂直领域 |
| 推理深度 | 3-5 层链式推理 | 支持多步因果推断 |
| 输入类型 | 结构化数据 + 自然语言 | 文本、数值、分类变量 |
| 输出格式 | 结构化 JSON | 包含结论、置信度、依据链 |
| 处理时间 | ≤ 5 秒/次 | 单次判断请求 |

### 2.3 设计原则

1. **可解释性优先**：每一条判断结论必须附带完整的推理路径
2. **模块化设计**：各组件低耦合、高内聚，支持独立迭代
3. **渐进式增强**：从规则驱动逐步过渡到数据驱动+规则融合
4. **人机协作**：模型辅助决策，最终判断权保留给人类专家

### 3.1 整体架构图（文字描述）

```
+------------------+       +-------------------+       +------------------+
|                  |       |                   |       |                  |
|  Input Layer     | ----> |  Processing Layer  | ----> |  Output Layer    |
|  (数据接入层)    |       |  (核心处理层)      |       |  (输出展示层)    |
|                  |       |                   |       |                  |
+------------------+       +-------------------+       +------------------+
        |                         |                            |
        v                         v                            v
  - 结构化数据            - 推理引擎                   - 结论展示
  - 文本描述              - 知识图谱                   - 推理路径
  - 上下文参数            - 规则匹配器                 - 置信度报告
                          - 证据聚合器                  - 建议列表
```

### 3.2 核心组件

| 组件名称 | 职责 | 技术选型（V1） |
|----------|------|----------------|
| **Input Parser** | 解析并标准化输入数据 | Python + Pydantic |
| **Knowledge Base** | 存储领域规则与事实 | 本地 JSON + YAML |
| **Inference Engine** | 执行链式推理与规则匹配 | 自定义推理引擎 |
| **Evidence Aggregator** | 合并多来源证据，计算置信度 | 加权融合算法 |
| **Output Formatter** | 格式化输出为结构化结果 | JSON Schema |
| **Logger & Monitor** | 记录推理过程与性能指标 | Python logging |

### 3.3 组件交互流程

```
[用户请求] 
    |
    v
[Input Parser] --> 校验与标准化
    |
    v
[Context Builder] --> 构建推理上下文
    |
    v
[Knowledge Base] --> 检索相关规则与事实
    |
    v
[Inference Engine] --> 执行推理（前向/后向链）
    |
    v

    |
    v
[Output Formatter] --> 结构化输出
    |
    v
[用户响应]
```

## 4. 专业判断层级体系

### 4.1 四层判断模型

M-BIZ 专业判断模型将判断过程分为四个递进层级：

```
Level 1: 事实判断 (Factual)
  - 客观事实的识别与提取
  - 示例：财务报表中的营收数据

Level 2: 规则判断 (Rule-based)
  - 基于预设规则的结论推导
  - 示例：若营收连续3季下降，则判定为"下行趋势"

Level 3: 分析判断 (Analytical)
  - 多维度交叉分析与模式识别
  - 示例：结合市场数据与内部指标的综合评估

Level 4: 策略判断 (Strategic)
  - 前瞻性建议与策略生成
  - 示例：基于当前判断给出行动建议
```

### 4.2 层级间的关系

- **向上依赖**：高层级判断依赖低层级判断提供的证据
- **向下约束**：高层级判断结果可反向验证低层级判断的一致性
- **循环迭代**：当新证据出现时，可从任意层级重新触发推理

## 5. 知识图谱与推理引擎

### 5.1 知识表示

采用 **三元组** 形式表示领域知识：

```
(实体1, 关系, 实体2, [置信度, 来源, 权重])
```

示例：

```
(营收下降, 导致, 利润下降, [0.9, "会计准则", 0.8])
(行业衰退, 导致, 营收下降, [0.7, "行业报告2025", 0.6])
```

### 5.2 规则引擎设计

#### 5.2.1 规则格式

```yaml
rules:
  - id: "R001"
    name: "营收趋势判断"
    condition:
      and:
        - fact: "营收连续下降季数"
          operator: ">="
          value: 3
        - fact: "同比下降幅度"
          operator: ">"
          value: 0.1
    conclusion:
      type: "rule"
      summary: "营收处于持续下行趋势"
      confidence: 0.85
    weight: 0.9
    tags: ["财务分析", "趋势判断"]
```

#### 5.2.2 推理策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **前向链** | 从已知事实出发，逐步推导结论 | 数据驱动的判断 |
| **后向链** | 从假设结论出发，反向验证所需证据 | 假设验证、目标导向判断 |
| **混合链** | 前向+后向结合，双向推理 | 复杂商业判断 |

### 5.3 证据聚合算法

#### 加权置信度计算

```
总置信度 = Σ(证据_i × 权重_i) / Σ(权重_i)
```

其中：
- 证据_i：单个证据的置信度 [0, 1]
- 权重_i：证据的相对重要性 [0, 1]

#### 冲突处理

当两个证据矛盾时：
1. 优先采用权重更高的证据
2. 若权重相近，取置信度平均值，并标记为"需人工复核"
3. 记录冲突证据对，供后续分析

## 6. 数据流与处理流程

### 6.1 完整处理流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  输入阶段  │ -> │  分析阶段  │ -> │  推理阶段  │ -> │  输出阶段  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │
     v               v               v               v
  数据收集       事实提取        规则匹配        结论生成
  参数校验       上下文构建      链式推理        置信度计算
  格式转换       特征计算        证据聚合        路径展示
```

### 6.2 输入规范

```json
{
  "request_id": "req_001",
  "query": {
    "type": "financial_analysis | risk_assessment | strategy_recommendation",
    "description": "判断该公司当前财务状况"
  },
  "data": {
    "structured": {
      "revenue": 1000000,
      "growth_rate": -0.15,
      "profit_margin": 0.05
    },
    "unstructured": "该公司近三个季度营收持续下滑..."
  },
  "context": {
    "industry": "technology",
    "region": "APAC",
    "time_period": "2026-Q2"
  },
  "config": {
    "min_confidence": 0.6,
    "max_depth": 5,
    "include_evidence_chain": true
  }
}
```

## 7. 评估与反馈机制

### 7.1 评估指标体系

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 判断准确率 | 正确判断数 / 总判断数 | ≥ 80% |
| 推理完整性 | 完整推理路径数 / 总判断数 | ≥ 90% |
| 平均处理时间 | 总处理时间 / 请求数 | ≤ 3000ms |
| 用户采纳率 | 被采纳建议数 / 总建议数 | ≥ 60% |
| 回溯覆盖率 | 可回溯判断数 / 总判断数 | ≥ 95% |

### 7.2 反馈闭环

```
[模型输出] --> [用户判断] --> [结果对比] --> [差异分析] --> [规则优化]
                                                                 |
                                                                 v
                                                          [知识库更新]
```

### 7.3 持续学习策略

1. **规则学习**：基于用户修正记录，自动生成候选规则
2. **权重优化**：根据历史准确率动态调整证据权重
3. **知识扩展**：识别新实体与新关系，扩展知识图谱
