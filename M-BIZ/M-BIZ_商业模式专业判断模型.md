## 1. 文档概述

### 1.1 文档目的

本文档定义 M-BIZ 商业模式专业判断模型（Business Model Judgment Model, BMJM），在通用专业判断模型（PJM）框架基础上，聚焦于商业模式的识别、分析、评估与优化建议，为商业模式诊断与设计提供系统化的判断能力。

### 1.2 适用范围

- **目标用户**：创业者、企业战略管理者、投资人、商业分析师
- **应用场景**：商业模式评估、竞争力诊断、创新机会识别、转型方案设计
- **领域范围**：涵盖九大商业模式构建模块，覆盖传统与数字化商业模式

### 1.3 与 PJM 的关系

| 项目 | 通用 PJM | 商业模式 BMJM |
|------|----------|---------------|
| 定位 | 通用判断框架 | 垂直领域专业模型 |
| 依赖 | 无 | 继承 PJM 架构与推理引擎 |
| 扩展 | 领域规则需自定义 | 预置商业模式领域知识库 |
| 输出 | 通用判断结论 | 商业模式专项诊断报告 |

## 2. 商业模式判断框架

### 2.1 判断总览

商业模式专业判断模型以 **Business Model Canvas（商业模式画布）** 为理论基础，结合 **VRIO 框架** 与 **价值驱动设计** 方法论，构建四维一体的判断体系：

| 维度 | 说明 | 方法 |
|------|------|------|
| 维度一：结构分析 | 九大模块逐一诊断 | Business Model Canvas |
| 维度二：健康度评估 | 财务与非财务指标综合评价 | 加权评分 + 规则判定 |
| 维度三：竞争力判断 | 资源与能力的竞争优势分析 | VRIO 框架 + 行业对标 |
| 维度四：进化建议 | 优化方向与策略生成 | 规则引擎 + 最佳实践匹配 |

### 2.2 判断层级映射

| PJM 层级 | 商业模式判断对应 | 示例 |
|----------|-----------------|------|
| L1: 事实判断 | 商业模式事实提取与描述 | "该企业采用订阅制收费模式" |
| L2: 规则判断 | 基于规则的模块健康度判定 | "客户获取成本高于行业均值2倍" |
| L3: 分析判断 | 多模块交叉分析与模式识别 | "高获客成本 + 低留存率 = 增长陷阱" |
| L4: 策略判断 | 商业模式优化建议与转型方向 | "建议从产品售卖转向服务订阅模式" |

### 2.3 商业模式类型体系

| 大类 | 子类型 | 典型特征 |
|------|--------|----------|
| 传统商业模式 | 产品制造型 | 实物产品生产与销售 |
| 传统商业模式 | 渠道分销型 | 多级代理与渠道网络 |
| 传统商业模式 | 零售服务型 | 终端消费者直接触达 |
| 传统商业模式 | 特许经营型 | 品牌授权+标准化运营 |
| 数字化商业模式 | 平台型 | 供需双方匹配撮合 |
| 数字化商业模式 | 订阅型 | 定期付费获取服务 |
| 数字化商业模式 | 免费增值型 | 基础免费+高级付费 |
| 数字化商业模式 | 按需服务型 | 即时响应按需付费 |
| 数字化商业模式 | 生态系统型 | 多业务协同锁定用户 |
| 混合/创新模式 | 硬件+服务型 | 硬件销售+持续服务费 |
| 混合/创新模式 | 数据驱动型 | 以数据为核心资产变现 |
| 混合/创新模式 | 社区驱动型 | 社区共识与价值共创 |
| 混合/创新模式 | 开放协作型 | 开源/开放API生态 |

## 3. 九维评估模型

### 3.1 九大维度定义

基于商业模式画布的九大构建模块：

| 编号 | 维度 | 核心问题 | 关键指标 |
|------|------|----------|----------|
| VP | 价值主张 | 为客户解决什么问题？ | 客户痛点匹配度、差异化程度 |
| CS | 客户细分 | 服务哪些客户群体？ | 细分市场明确度、市场容量 |
| CH | 渠道通路 | 如何触达客户？ | 渠道覆盖度、获客效率 |
| CR | 客户关系 | 如何维系客户？ | 留存率、复购率、NPS |
| RS | 收入来源 | 如何获得收入？ | 收入结构多样性、ARPU |
| KR | 核心资源 | 需要哪些核心资产？ | 资源独特性、可替代性 |
| KA | 关键业务 | 最关键的业务活动？ | 核心能力聚焦度、运营效率 |
| KP | 重要伙伴 | 需要哪些合作伙伴？ | 供应链稳定性、合作深度 |
| CS | 成本结构 | 主要成本构成？ | 成本效率、规模效应 |

### 3.2 维度评分标准

每个维度采用 **1-5 分** 评分制：

| 分数 | 等级 | 描述 |
|------|------|------|
| 5 | 卓越 | 远超行业标准，构成核心竞争力 |
| 4 | 良好 | 超过行业平均水平，有竞争优势 |
| 3 | 合格 | 达到行业平均水平，无显著短板 |
| 2 | 薄弱 | 低于行业平均，需重点关注 |
| 1 | 缺失/危险 | 严重不足或缺失，构成重大风险 |

### 3.3 健康度综合判定

**综合健康度公式：**

综合健康度 = (各维度加权得分之和) / 5 x 100%

**V1 预置维度权重：**

| 维度 | VP | CS | CH | CR | RS | KR | KA | KP | CS |
|------|----|----|----|----|----|----|----|----|----|
| 权重 | 0.15 | 0.10 | 0.10 | 0.10 | 0.15 | 0.10 | 0.10 | 0.05 | 0.15 |

**健康度等级：**

| 范围 | 等级 | 含义 |
|------|------|------|
| >= 85% | 健康 | 商业模式整体良好 |
| >= 65% | 亚健康 | 存在部分短板需关注 |
| >= 45% | 警示 | 多个维度存在问题 |
| < 45% | 危险 | 商业模式存在重大缺陷 |

## 4. 判断规则库

### 4.1 规则分类

| 规则类别 | 数量(V1) | 说明 |
|----------|-----------|------|
| 结构完整性规则 | 15+ | 检查九大模块是否完整定义 |
| 一致性规则 | 20+ | 检查模块间逻辑一致性 |
| 健康度阈值规则 | 25+ | 基于指标的阈值判定 |
| 模式识别规则 | 10+ | 识别典型商业模式模式 |
| 风险预警规则 | 15+ | 识别潜在商业模式风险 |

### 4.2 规则定义格式

```yaml
# 规则 R-BM-001: 收入来源单一性风险
- id: "R-BM-001"
  name: "收入来源单一性风险判定"
  domain: "BM.RS"
  level: 2
  condition:
    and:
      - metric: "revenue_streams_count"
        operator: "<="
        value: 1
      - metric: "top_revenue_share"
        operator: ">"
        value: 0.8
  conclusion:
    type: "risk_warning"
    summary: "收入来源高度单一，抗风险能力弱"
    severity: "high"
    confidence: 0.85
  suggestion: "建议拓展多元化收入渠道，降低单来源依赖风险"
```

```yaml
# 规则 R-BM-002: 价值主张-客户匹配度
- id: "R-BM-002"
  name: "价值主张-客户细分匹配度"
  domain: "BM.VP"
  level: 3
  condition:
    and:
      - fact: "value_proposition_definition"
        operator: "exists"
        value: true
      - fact: "customer_pain_point_coverage"
        operator: ">="
        value: 0.7
      - fact: "target_segment_clarity"
        operator: ">="
        value: 3
  conclusion:
    type: "positive"
    summary: "价值主张与客户需求高度匹配"
    confidence: 0.8
```

## 5. 行业基准与对标

### 5.1 基准数据结构

每个行业基准包含核心指标的平均值、上四分位与下四分位：

```json
{
  "industry_benchmark": {
    "industry": "saas",
    "region": "global",
    "year": 2025,
    "metrics": {
      "monthly_churn_rate": {
        "avg": 0.05,
        "top_quartile": 0.02,
        "bottom_quartile": 0.10
      },
      "cac": {
        "avg": 500,
        "top_quartile": 200,
        "bottom_quartile": 1200
      },
      "ltv_cac_ratio": {
        "avg": 3.5,
        "top_quartile": 5.0,
        "bottom_quartile": 1.5
      },
      "gross_margin": {
        "avg": 0.75,
        "top_quartile": 0.85,
        "bottom_quartile": 0.55
      },
      "nps": {
        "avg": 40,
        "top_quartile": 60,
        "bottom_quartile": 20
      }
    }
  }
}
```

### 5.2 偏离度计算

偏离度 = (企业值 - 基准值) / 基准值

判定标准:
- 偏离度 in [-0.1, 0.1]: 与行业持平
- 偏离度 > 0.1: 优于行业(正向指标) / 劣于行业(负向指标)
- 偏离度 < -0.1: 劣于行业(正向指标) / 优于行业(负向指标)

## 7. 判断流程与示例

### 7.1 完整判断流程

```
Step 1: 信息采集
  - 收集企业基本信息、财务数据、运营数据
  - 结构化问卷或 API 输入
        |
        v
Step 2: 模块评分
  - 按九大维度逐一评估打分
  - 结合规则库自动判定
        |
        v
Step 3: 综合评估
  - 计算综合健康度
  - 执行一致性检查
  - 模式匹配与画像识别
        |
        v
Step 4: 对标分析
  - 匹配行业基准
  - 计算偏离度
  - 识别差距与优势
        |
        v
Step 5: 策略建议
  - 基于规则生成改进建议
  - 每条建议自动附加可执行的验证动作（verification_action）
  - 输出结构化判断报告（每条 recommendation 必须包含 verification_action）

### 7.4 "不算死账"原则与验证动作机制

#### 7.4.1 原则定义

"不算死账"是 BMJM 的核心输出约束：**每一条 strategic_suggestion 必须附带可执行的验证动作（verification_action）**，即告诉用户"需要做什么来验证这条建议是否可行"，而非给出无法被验证的"死结论"。

错误示例（死账）：
  建议："建议开拓海外市场"
  问题：无法验证，没有可操作的下一步

正确示例（活账）：
  建议："建议开拓东南亚市场"
  verification_action: "请先收集目标国家前 3 大竞争对手的定价与市场份额数据，再做决定"

#### 7.4.2 验证动作自动生成机制

推理引擎在输出 strategic_suggestions 前，执行以下强制流程：

```
[规则引擎输出建议]
        |
        v
[验证动作匹配器]
  - 根据建议类型匹配预置验证动作模板
  - 若匹配失败，进入验证动作生成器
        |
        v
[验证动作生成器]
  - 解析建议的目标维度与操作类型
  - 结合输入数据中的具体指标值
  - 输出可执行的验证步骤
        |
        v
[合规检查器]
  - 检查 verification_action 是否存在
  - 检查是否为"死账"（空洞/无法验证/缺证据链/推诿责任）
  - 若不合规则拦截并警告
        |
        v
[输出结果]
```

#### 7.4.3 验证动作模板库（V1 预置）

| 建议类型 | 验证动作模板 |
|----------|--------------|
| 拓展新收入渠道 | "请补充 {n} 个月的真实 {业务类型} 流水数据再决定" |
| 调整定价策略 | "请对 top {n} 客户做付费意愿调研，若接受率 > {threshold}% 再推进" |
| 优化获客渠道 | "请试运行 {渠道} {n} 个月，对比 CAC 和转化率后再决策" |
| 产品功能扩展 | "请先对 {n} 名目标用户做可用性测试，任务完成率 > {threshold}% 再投入开发" |
| 引入合作伙伴 | "请先联系 {n} 家潜在伙伴做 PoC，验证集成可行性与 ROI" |
| 拓展新市场 | "请先收集目标市场的 {n} 项关键数据（竞争/监管/规模），再做决策" |
| 组织调整 | "请先分析过去 {n} 个月的团队效能数据，识别瓶颈后再设计方案" |

#### 7.4.4 合规检查标准

| 检查项 | 标准 | 违规处理 |
|--------|------|----------|
| 存在性 | verification_action 字段非空 | 拦截并自动生成 |
| 可执行性 | 包含可量化的步骤与判断标准 | 标记为"需人工审核" |
| 时限合理性 | 验证周期不超过 3 个月 | 标记为"建议拆分验证步骤" |
| 数据可及性 | 验证所需数据为企业合理可得 | 标记为"数据来源需确认" |
| 结论可证伪 | 验证结果可明确判定可行/不可行 | 拦截并要求重写 |

#### 7.4.5 规则中的 verification_action 字段

每个规则的 conclusion 中新增 verification_action 模板字段：

```yaml
- id: "R-BM-001"
  name: "收入来源单一性风险判定"
  conclusion:
    type: "risk_warning"
    summary: "收入来源高度单一，抗风险能力弱"
    severity: "high"
    confidence: 0.85
  suggestion: "建议拓展多元化收入渠道，降低单来源依赖风险"
  verification_action_template: "请补充 {months} 个月的真实 {revenue_type} 流水数据，分析各渠道收入占比后再决定拓展方向"
```

## 8. 与 M-BIZ 体系的集成

### 8.1 集成关系

BMJM 继承 PJM 的推理引擎与架构，同时调用 KAS 中的商业模式知识包：

| 上游组件 | 集成方式 | 提供能力 |
|----------|----------|----------|
| M-BIZ PJM | 架构继承 | 推理引擎、证据聚合、输出格式化 |
| M-BIZ KAS | 知识包调用 | 规则库、行业基准、商业模式画像 |

### 8.2 知识资产映射

| 知识包 | 用途 | 依赖版本 |
|--------|------|----------|
| biz-model-core | 九维评估模型定义与规则 | >= 1.0 |
| biz-model-benchmark | 行业基准数据 | >= 1.0 |
| biz-model-profiles | 典型商业模式画像库 | >= 1.0 |
| biz-model-cases | 历史案例库 | >= 1.0 |

### 8.3 扩展接口

BMJM 提供以下扩展点供上层调用或定制：

| 扩展点 | 说明 | 配置方式 |
|--------|------|----------|
| 自定义权重 | 调整九大维度的权重分配 | YAML 配置文件 |
| 自定义规则 | 添加企业特定或行业特定的判断规则 | 规则文件注入 |
| 自定义基准 | 上传私有行业基准数据 | API / 文件导入 |
| 自定义画像 | 注册新的商业模式画像 | JSON 描述文件 |
| 报告模板 | 定制输出报告的格式与内容 | 模板引擎配置 |
