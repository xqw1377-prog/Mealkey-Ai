## 1. 文档概述

### 1.1 文档目的

本文档定义 M-BIZ 知识资产体系（Knowledge Asset System, KAS）的整体架构、分类模型、生命周期管理及与专业判断模型的集成方案，为知识资产的构建、存储、维护与应用提供统一的设计规范。

### 1.2 适用范围

- **目标用户**：领域专家、知识工程师、系统开发者
- **覆盖范围**：结构化知识、半结构化规则、非结构化文档、经验知识
- **系统边界**：知识采集 → 建模 → 存储 → 检索 → 应用 → 迭代

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **知识资产** | 经过提炼、验证并结构化表达的领域知识单元 |
| **知识元** | 知识资产的最小不可分割单元，如一条规则、一个事实 |
| **知识包** | 按主题或领域聚合的知识资产集合 |
| **知识图谱** | 以实体-关系-实体三元组形式组织的知识网络 |
| **知识版本** | 知识资产在特定时间点的快照状态 |
| **知识置信度** | 对知识内容准确性与可靠性的度量，取值范围 [0, 1] |

## 3. 知识资产分类模型

### 3.1 五维分类体系

```
+------------------+
|  知识资产分类体系  |
+------------------+
         |
   +-----+-----+-----+-----+-----+
   |     |     |     |     |     |
   v     v     v     v     v     v
 形态    领域    结构    来源    可信度
```

#### 维度一：知识形态 (Form)

| 形态 | 编码 | 说明 | 示例 |
|------|------|------|------|
| 事实型 | FCT | 客观陈述、不可辩驳的基本事实 | "2025年公司营收为1.2亿" |
| 规则型 | RUL | 条件-结论形式的判断规则 | "若毛利率<20%，则盈利能力较弱" |
| 概念型 | CPT | 领域术语、定义与概念解释 | "EBITDA = 息税折旧摊销前利润" |
| 流程型 | PRC | 步骤化、流程化的操作知识 | "财务报表分析五步法" |
| 经验型 | EXP | 来自实践的隐性知识提炼 | "通常连续3季度下降需启动预警" |
| 案例型 | CAS | 历史案例与最佳实践 | "2023年某公司转型成功案例" |

#### 维度二：领域分类 (Domain)

```
Business (BIZ)
  ├── 财务分析 (FIN)
  ├── 市场营销 (MKT)
  ├── 运营管理 (OPS)
  ├── 人力资源 (HR)
  ├── 战略规划 (STR)
  ├── 风险管理 (RSK)
  ├── 合规审查 (CMP)
  └── 行业知识 (IND)
```

#### 维度三：结构程度 (Structure)

| 级别 | 说明 | 存储方式 |
|------|------|----------|
| S1 - 高结构化 | 明确的字段、类型、关系 | 图数据库节点/边 |
| S2 - 半结构化 | 模板化但内容灵活 | JSON/YAML 文档 |
| S3 - 低结构化 | 自由文本、图表、多媒体 | 文档存储 |

#### 维度四：来源分类 (Source)

| 来源 | 编码 | 可靠度基准 |
|------|------|------------|
| 领域专家 | EXPERT | 0.9 |
| 行业标准/法规 | STANDARD | 0.95 |
| 学术文献 | ACADEMIC | 0.85 |
| 企业内部数据 | INTERNAL | 0.8 |
| 公开报告/资讯 | PUBLIC | 0.7 |
| 模型推理产出 | INFERRED | 0.6 |

#### 维度五：可信度等级 (Confidence)

| 等级 | 范围 | 标签 | 处理策略 |
|------|------|------|----------|
| L1 | [0.9, 1.0] | 高可信 | 直接使用 |
| L2 | [0.7, 0.9) | 中可信 | 需附带来源引用 |
| L3 | [0.5, 0.7) | 低可信 | 需人工复核后使用 |
| L4 | [0, 0.5) | 待验证 | 不建议直接使用 |

## 4. 资产生命周期管理

### 4.1 生命周期阶段

```
[创建] --> [评审] --> [发布] --> [使用] --> [维护] --> [归档/废弃]
   |          |          |          |          |
   v          v          v          v          v
  采集与建模  专家审核  上线部署  持续监控  定期更新
```

### 4.2 状态流转图

```
+---------+      +---------+      +---------+
|  草稿    | --> |  评审中  | --> |  已批准  |
| (Draft)  |     | (Review) |     | (Approved)|
+---------+      +---------+      +---------+
    |                              |
    | (驳回)                       | (过期)
    v                              v
+---------+      +---------+      +---------+
|  已拒绝   |     |  已发布  | --> |  已归档  |
| (Rejected)|     | (Published)|   | (Archived)|
+---------+      +---------+      +---------+
                     |
                     | (更新)
                     v
                  +---------+
                  |  修订中  |
                  | (Revising)|
                  +---------+
```

### 4.3 版本管理策略

| 版本号格式 | 说明 | 示例 |
|-----------|------|------|
| **X.Y.Z** | 主版本.次版本.修订号 | 1.3.2 |
| 主版本 X | 重大重构或不兼容变更 | 1.x → 2.x |
| 次版本 Y | 新增知识或功能增强 | 1.2 → 1.3 |
| 修订号 Z | 修正错误或微调 | 1.3.1 → 1.3.2 |

**版本兼容规则：**

- 同一主版本下，知识资产可向后兼容
- 主版本变更需同步更新依赖该资产的所有规则与模型
- 每个版本需记录变更日志（CHANGELOG）

### 4.4 废弃策略

| 废弃原因 | 处理方式 | 保留期限 |
|----------|----------|----------|
| 知识过时 | 标记为 "DEPRECATED"，停止推荐 | 保留1年供回溯 |
| 被更优知识替代 | 建立替代关系链 | 保留6个月 |
| 确认错误 | 立即废弃并触发相关资产复核 | 永久保留错误记录 |

## 5. 知识图谱结构设计

### 5.1 图模型定义

#### 节点类型 (Node Types)

| 节点标签 | 说明 | 属性 |
|----------|------|------|
| `Entity` | 业务实体 | id, name, type, description |
| `Concept` | 领域概念 | id, name, definition, domain |
| `Rule` | 判断规则 | id, name, condition, conclusion, confidence |
| `Case` | 案例 | id, title, description, outcome, tags |
| `Metric` | 指标 | id, name, formula, unit, threshold |
| `Document` | 参考文档 | id, title, source, content_uri |

#### 关系类型 (Relationship Types)

| 关系标签 | 说明 | 属性 |
|----------|------|------|
| `HAS_PROPERTY` | 实体拥有属性 | confidence, source |
| `RELATES_TO` | 关联关系 | relation_type, weight |
| `CAUSES` | 因果关系 | strength, latency |
| `DERIVES_FROM` | 推导来源 | formula, params |
| `SUPPORTS` | 支持证据 | relevance, confidence |
| `CONTRADICTS` | 矛盾关系 | severity, resolution |
| `REFERENCES` | 引用关系 | context, page |

### 5.2 示例图谱结构

```
[营收下降] --(CAUSES)--> [利润下降]
     |                       |
  (HAS_PROPERTY)         (HAS_PROPERTY)
     |                       |
  [下降幅度:15%]         [下降幅度:8%]
     |
  (RELATES_TO)
     |
[行业衰退] --(SUPPORTS)--> [市场报告2025]
```

### 5.3 核心图谱索引要求

| 索引类型 | 字段 | 用途 |
|----------|------|------|
| 唯一索引 | id | 快速定位资产 |
| 全文索引 | name, description | 关键词搜索 |
| 范围索引 | confidence, version | 筛选查询 |
| 全文索引 | content | 语义搜索 |
| 标签索引 | tags, domain | 分类导航 |

## 7. 知识质量管理

### 7.1 质量评估维度

| 维度 | 定义 | 评估方法 | 权重 |
|------|------|----------|------|
| **准确性** | 知识与事实的符合程度 | 专家审核 + 交叉验证 | 35% |
| **完整性** | 知识覆盖范围与深度 | 覆盖率分析 + 缺口识别 | 20% |
| **一致性** | 知识之间无矛盾 | 自动冲突检测 | 20% |
| **时效性** | 知识的有效性与新鲜度 | 时间戳 + 行业变化跟踪 | 15% |
| **可用性** | 知识被检索与使用的便捷度 | 使用率统计 + 用户反馈 | 10% |

### 7.2 质量评分公式

```
Q = 0.35 × A + 0.20 × C + 0.20 × S + 0.15 × T + 0.10 × U

其中：
  A = 准确性得分 [0, 1]
  C = 完整性得分 [0, 1]
  S = 一致性得分 [0, 1]
  T = 时效性得分 [0, 1]
  U = 可用性得分 [0, 1]
```

### 7.3 质量门禁

| 质量等级 | Q 值范围 | 发布策略 |
|----------|----------|----------|
| 优秀 | [0.9, 1.0] | 可直接发布使用 |
| 良好 | [0.7, 0.9) | 建议补充说明后发布 |
| 一般 | [0.5, 0.7) | 需更新后重新评审 |
| 不合格 | [0, 0.5) | 禁止发布，返回修订 |

### 7.4 冲突检测机制

#### 冲突类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 直接矛盾 | 两个规则结论互斥 | R1: "A→B", R2: "A→¬B" |
| 间接矛盾 | 通过推理链产生矛盾 | R1: "A→B", R2: "B→C", R3: "A→¬C" |
| 边界冲突 | 条件边界重叠导致歧义 | "x>10→高", "x<15→低" (x=12 时冲突) |

#### 检测流程

```
[规则/知识入库]
      |
      v
[冲突检测引擎]
      |
      +--> [直接矛盾检测] --> 匹配相同条件+不同结论
      |
      +--> [间接矛盾检测] --> 推理链推导验证
      |
      +--> [边界重叠检测] --> 条件区间交叉分析
      |
      v
[冲突报告] --> [人工裁决] --> [知识更新]
```

## 8. 资产访问与控制

### 8.1 访问控制模型

采用 **RBAC（基于角色的访问控制）** 模型：

| 角色 | 权限范围 | 适用对象 |
|------|----------|----------|
| `admin` | 完全控制（CRUD + 发布 + 系统配置） | 系统管理员 |
| `knowledge_engineer` | 创建、编辑、删除知识资产 | 知识工程师 |
| `domain_expert` | 创建、编辑自己领域资产，审核 | 领域专家 |
| `reviewer` | 查看、评论、审批/驳回 | 资深专家 |
| `reader` | 只读访问 | 普通用户 |
| `system` | API 访问权限（供判断模型调用） | 服务账户 |

### 8.2 权限矩阵

| 操作 | admin | knowledge_engineer | domain_expert | reviewer | reader | system |
|------|-------|--------------------|---------------|----------|--------|--------|
| 查看 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 创建 | ✅ | ✅ | ✅ (领域内) | ❌ | ❌ | ❌ |
| 编辑 | ✅ | ✅ | ✅ (领域内) | ❌ | ❌ | ❌ |
| 删除 | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| 审核 | ✅ | ❌ | ✅ (领域内) | ✅ | ❌ | ❌ |
| 发布 | ✅ | ❌ | ❌ | ✅ | ❌ | ❌ |
| API 调用 | ✅ | ✅ | ✅ | ✅ | ❌ | ✅ |

### 8.3 审计日志

所有知识资产的操作需记录审计日志：

## 9. 与专业判断模型的集成

### 9.1 集成关系

```
+------------------+              +------------------+
|  专业判断模型      |              |  知识资产体系      |
|  (PJM)           |  <--------->  |  (KAS)            |
+------------------+              +------------------+
        |                                 |
        | ① 请求知识检索                     |
        |--------------------------------->|
        |                                 |
        | ② 返回匹配规则与事实                |
        |<---------------------------------|
        |                                 |
        | ③ 上报推理结果与新发现              |
        |--------------------------------->|
        |                                 |
        | ④ 反馈知识质量评分                 |
        |<---------------------------------|
```

### 9.2 接口定义

#### 知识检索接口

```
GET /api/v1/knowledge/search

Parameters:
  - query: str              # 查询内容
  - domain: str (optional)  # 限定领域
  - form: str (optional)    # 限定知识形态
  - min_confidence: float   # 最低置信度 (默认 0.6)
  - limit: int              # 返回数量 (默认 10)

Response:
{
  "results": [
    {
      "asset_id": "KAS-...",
      "name": "...",
      "form": "RUL",
      "domain": "BIZ.FIN",
```

### 9.3 知识回流机制

PJM 在使用 KAS 知识完成判断后，应将以下信息回流至知识资产体系：

1. **推理轨迹**：哪些知识资产被触发及其使用频率
2. **准确率统计**：知识资产支撑的判断与实际结果的一致性
3. **知识缺口**：推理过程中发现但知识库中缺失的知识
4. **新候选知识**：推理过程中生成的、经过验证的新规则
