## 1. 文档概述

### 1.1 文档目的

本文档定义 MealKey 产品的 Founder Intelligence Layer（创始人智能层）的核心认知架构——**创业认知链**（Entrepreneurial Cognition Chain, ECC）。该认知链是 M-BIZ 专业判断模型家族在"创业者决策辅助"这一垂直场景中的具象化产物，将通用商业判断能力转化为面向创始人的、可交互的、带验证闭环的认知流。

### 1.2 适用对象

| 角色 | 关注重点 |
|------|----------|
| 产品经理 | 认知链的产品形态、用户体验设计 |
| 算法工程师 | 链式推理引擎实现、规则配置 |
| 创业者（终端用户） | 能获得什么样的认知服务 |
| 投资人/顾问 | 认知链输出的可信度与可验证性 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **认知链（ECC）** | 从事实到验证的五层递进推理链路，是 Founder Intelligence Layer 的核心推理范式 |
| **认知节点** | 认知链中的单个推理步骤，包含输入、规则引用、输出与置信度 |
| **验证动作** | 策略层输出的可执行验证步骤，用户完成验证后结果回注认知链 |
| **认知回注** | 用户执行验证动作后，将真实结果写回认知链形成闭环的过程 |
| **「不算死账」** | 核心约束：认知链不允许输出无法被验证的"死结论" |

### 1.4 与 M-BIZ 体系的关系

```
M-BIZ Professional Judgment Model (PJM)
  └── 通用判断框架：四层递进（事实→规则→分析→策略）
        |
        v
M-BIZ Business Model Judgment Model (BMJM)
  └── 垂直领域模型：九维评估 + 商业模式画像
        |
        v
MealKey Founder Intelligence Layer (FIL)
  └── 创业场景具象化：五层认知链 + 验证闭环 + 不算死账
```

| 继承关系 | M-BIZ PJM | BMJM | MealKey FIL |
|----------|-----------|------|-------------|
| 判断层级 | 4 层 (L1-L4) | 4 层 | **5 层**（新增 L5 验证认知）|
| 领域范围 | 通用商业 | 商业模式 | 创业全生命周期 |
| 输出约束 | 无强制 | verification_action | **验证动作即出口，无验证不输出** |
| 反馈闭环 | 可选反馈 | 建议反馈 | **强制回注，验证结果写回知识库** |

## 2. 从 M-BIZ 到 Founder Intelligence Layer

### 2.1 体系演进路线

```
M-BIZ PJM (通用判断)
     │
     ├── 知识资产体系 KAS (知识存储)
     │
     ├── BMJM (商业模式垂直)
     │
     └── MealKey FIL (创业认知链)
           │
           ├── 继承 PJM 推理引擎
           ├── 继承 BMJM 九维评估 + 画像库
           ├── 继承 KAS 知识资产管理
           ├── 继承「不算死账」验证动作机制
           │
           └── 新增：
                ├── L5 验证认知层
                ├── 认知回注机制
                ├── 创业阶段感知
                └── MealKey 场景适配器
```

### 2.2 MealKey FIL 在 M-BIZ 体系中的定位

| 维度 | M-BIZ 定位 | MealKey FIL 定位 |
|------|-----------|------------------|
| 用户 | 商业分析师、决策者 | 早期创业者（非专业背景） |
| 输入 | 结构化企业数据 | 碎片化、不完整、创始人自述 |
| 输出 | 结构化判断报告 | 可对话的认知流 + 可执行的验证清单 |
| 交互 | API / 批处理 | 对话式 + 渐进式追问 |
| 闭环 | 可选 | **强制回注** |

### 2.3 创始人的认知痛点与 FIL 的对应解法

| 创始人痛点 | FIL 解法 |
|-----------|----------|
| 信息碎片化，看不清全局 | L1 事实认知 → 结构化梳理 |
| 凭直觉决策，缺少规则参照 | L2 规则认知 → 行业规则匹配 |
| 单点分析，忽略交叉影响 | L3 分析认知 → 多维度交叉推理 |
| 知道问题但不知道第一步做什么 | L4 策略认知 → 可执行建议 |
| 做了决策但无法验证对错 | L5 验证认知 → 验证动作 + 结果回注 |

## 3. 创业认知链总览

### 3.1 五层认知链架构

```
┌─────────────────────────────────────────────────────────┐
│                   创业认知链 (ECC)                        │
│                                                         │
│   L5: 验证认知 (Verification) ──── 回注 → 知识库       │
│     ↑                                                    │
│   L4: 策略认知 (Strategy) ──── 带验证动作的建议         │
│     ↑                                                    │
│   L3: 分析认知 (Analysis) ──── 交叉推理 + 模式识别     │
│     ↑                                                    │
│   L2: 规则认知 (Rule) ──────── 规则匹配 + 基准对标     │
│     ↑                                                    │
│   L1: 事实认知 (Fact) ──────── 信息收集 + 结构化        │
│                                                         │
│   ┌─────────────────────────────────────────────────┐   │
│   │   Input: 创始人自述 / 问卷 / 公开数据 / API     │   │
│   └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 3.2 认知链的核心特征

| 特征 | 说明 |
|------|------|
| **递进性** | 每层依赖下一层的输出作为输入，形成推理链条 |
| **可回溯** | 任意结论都可追溯到 L1 的事实依据 |
| **可验证** | L5 验证认知是所有认知的强制出口 |
| **可回注** | 验证结果写回知识库，优化后续推理 |
| **渐进式** | 当 L1 信息不足时，触发追问而非强行推理 |

### 3.3 认知链的触发场景

| 场景 | 触发点 | 认知链深度 |
|------|--------|-----------|
| 创始人在产品中提问 | "我的商业模式行不行？" | L1→L5 完整链路 |
| 月度健康度检查 | 定时触发 | L1→L3 快速扫描 |
| 验证结果回注 | 用户完成验证动作 | L5→知识库 单步 |
| 投资人提问 | "你们的关键风险是什么？" | L2→L4 定向推理 |
| 竞品动态监控 | 外部数据更新 | L1→L3 增量更新 |

## 4. 认知层 1：事实认知

### 4.1 定位

事实认知是认知链的基础层，负责将创始人提供的碎片化信息转化为结构化的认知节点。该层不做判断，只做梳理与确认。

### 4.2 输入

| 输入类型 | 示例 | 来源 |
|----------|------|------|
| 创始人自述 | "我们做 AI 客服，客户是中小电商" | 对话/问卷 |
| 业务数据 | 营收、客户数、客单价 | 手动填写/API |
| 文档上传 | BP、财务表格、用户调研报告 | 文件上传 |
| 公开信息 | 行业报告、竞品动态 | 数据抓取 |

### 4.3 认知节点

认知节点是事实认知层的最小输出单元，每个节点对应一个可被后续认知层引用的"事实陈述"。

```json
{
  "node_id": "FACT-001",
  "type": "fact_statement",
  "category": "value_proposition",
  "statement": "目标客户是中小型电商企业",
  "confidence": 0.8,
  "source": "创始人自述",
  "needs_verification": true,
  "verification_status": "unverified",
  "follow_up_questions": [
    "目标客户的平均月营收范围是多少？",
    "目前已经触达了多少家目标客户？"
  ],
  "created_at": "2026-07-10T12:00:00Z"
}
```

### 4.4 追问机制

当信息不足以构建完整的事实基底时，触发追问：

```
创始人: "我做一个 AI 客服产品"
   ↓
L1 解析: 事实节点缺失（客户是谁？痛点是啥？）
   ↓
追问: "这个 AI 客服主要帮谁解决什么问题？能举一个具体的例子吗？"
   ↓
创始人: "中小电商的客服成本太高了，我们帮他们降本"
   ↓
L1 完成: 客户=中小电商，痛点=客服成本高，价值=降本
   ↓
进入 L2
```

### 4.5 输出到 L2

事实认知层将结构化的事实节点打包传递给 L2 规则认知层：

```json
{
  "layer": "L1",
  "status": "completed",
  "fact_nodes": [
    { "id": "FACT-001", "category": "customer_segment", "statement": "中小型电商企业" },
    { "id": "FACT-002", "category": "pain_point", "statement": "客服成本高，响应效率低" },
    { "id": "FACT-003", "category": "value_proposition", "statement": "AI 驱动的全渠道智能客服" },
    { "id": "FACT-004", "category": "revenue_model", "statement": "按月订阅付费" }
  ],
  "missing_categories": ["key_metrics", "competitive_landscape"],
  "overall_confidence": 0.75,
  "recommend_action": "proceed_to_L2"
}
```

## 5. 认知层 2：规则认知

### 5.1 定位

规则认知层将 L1 的事实节点与预置规则库进行匹配，输出规则级别的判断结论。该层继承 BMJM 的判断规则库，并扩展了创业阶段的特有规则。

### 5.2 规则匹配流程

```
L1 事实节点
    |
    v
[规则索引] → 匹配 domain + 条件
    |
    v
[规则执行] → 输入事实值，计算结论
    |
    v
[置信度计算] → 结合规则 weight 与事实 confidence
    |
    v
L2 规则认知输出
```

### 5.3 创业阶段的规则扩展

MealKey FIL 在 BMJM 规则基础上，新增创业阶段感知规则：

| 阶段 | 规则重点 | 示例规则 |
|------|----------|----------|
| 种子期 (Seed) | 价值主张验证 | "如果目标客户定义模糊（<2个明确特征），则 PMF 风险高" |
| 成长期 (Growth) | 规模化效率 | "如果 CAC > LTV/3，则单位经济模型不可持续" |
| 成熟期 (Mature) | 竞争壁垒 | "如果核心资源可替代性评分 > 3，则竞争壁垒薄弱" |
| 转型期 (Pivot) | 转型方向验证 | "如果连续两个季度核心指标恶化，建议触发转型评估" |

### 5.4 L2 输出结构

```json
{
  "layer": "L2",
  "status": "completed",
  "rule_judgments": [
    {
      "rule_id": "R-BM-001",
      "domain": "BM.RS",
      "input_facts": ["FACT-004"],
      "conclusion": "收入来源单一（仅月订阅），抗风险能力弱",
      "confidence": 0.85,
      "severity": "warning"
    },
    {
      "rule_id": "R-ECC-SEED-001",
      "domain": "ECC.PMF",
      "input_facts": ["FACT-001", "FACT-002"],
      "conclusion": "价值主张与客户痛点匹配度较高，PMF 基础良好",
      "confidence": 0.75,
      "severity": "positive"
    }
  ],
  "risk_count": 1,
  "positive_count": 1,
  "recommend_action": "proceed_to_L3",
  "trigger_L3_dimensions": ["RS", "CH"]
}
```

## 6. 认知层 3：分析认知

### 6.1 定位

分析认知层对 L2 输出的规则判断进行多维度交叉分析，识别模式、发现矛盾、评估综合健康度。该层继承 BMJM 的九维评估方法，并扩展了创业场景特有的交叉分析模式。

### 6.2 交叉分析模式

| 模式 | 说明 | 触发条件 |
|------|------|----------|
| 一致性检查 | 检查多规则结论之间是否存在矛盾 | L2 输出中包含冲突标记 |
| 模式识别 | 将当前事实匹配到已知的商业模式画像 | L1 + L2 数据完整度 > 60% |
| 因果链分析 | 识别"因→果"关系链 | 存在 3 个以上关联域规则 |
| 短板效应识别 | 找到健康度最低的维度及其连锁影响 | 存在评分 ≤ 2 的维度 |
| 增长杠杆识别 | 识别投入产出比最高的改进方向 | 综合健康度 < 80% |

### 6.3 创业阶段适配的分析权重

| 阶段 | 高权重维度 | 低权重维度 |
|------|-----------|-----------|
| Seed | VP(0.25), CS(0.20), KR(0.15) | KP(0.05), CS(0.10) |
| Growth | CH(0.20), CR(0.20), RS(0.20) | VP(0.10), KR(0.10) |
| Mature | KR(0.20), KP(0.15), CS(0.20) | VP(0.05), CS(0.05) |
| Decline/Pivot | VP(0.25), RS(0.20), KA(0.20) | CH(0.05), CR(0.05) |

```json
    "level": "sub_healthy",

      "RS": 3, "KR": 3, "KA": 3, "KP": 3, "CS": 3
    }

  "pattern_matches": [
    {
      "pattern": "saas_subscription",

      "matched_dimensions": ["VP", "RS", "KA"]

    "consistency_check": "passed",
    "key_weakness": "CH(渠道)评分最低，与 RS(收入)的单一性形成复合风险",

  },

}
```

## 7. 认知层 4：策略认知

### 7.1 定位

策略认知层基于 L3 的分析结论，生成可执行的策略建议。**该层受到「不算死账」原则的强制约束**：每一条策略建议必须附带验证动作（verification_action），不允许输出无法被验证的"死建议"。

### 7.2 策略生成流程

```
L3 分析结论（短板、杠杆、风险）
    |
    v
[策略匹配器]
  - 按短板维度匹配策略模板
  - 按增长杠杆匹配优先级
    |
    v
[验证动作生成器]
  - 根据建议类型匹配验证动作模板
  - 模板参数注入：阶段、规模、具体指标
    |
    v
[「不算死账」合规检查]
  - verification_action 非空
  - 验证步骤可执行
  - 验证周期 ≤ 3 个月
  - 结论可证伪
    |
    v
[优先级排序]
  - High: 影响核心生存的短板
  - Medium: 影响增长效率的优化
  - Low: 锦上添花
    |
    v
L4 策略认知输出
```

### 7.3 验证动作模板（MealKey 版）

| 建议类型 | 模板 | 示例 |
|----------|------|------|
| 获客渠道优化 | "请试运行 {渠道} {n} 个月，统计 CAC 与转化率" | "请试运行小红书内容营销 1 个月，统计 CAC 与转化率" |
| 收入模式调整 | "请对 top {n} 客户做付费意愿调研" | "请对 top 20 客户做付费意愿调研，接受率 > 30% 再推进" |
| 产品功能扩展 | "请先对 {n} 名用户做内测" | "请先对 10 名种子用户做内测，NPS > 40 再上线" |
| 客户留存优化 | "请提取过去 {n} 个月流失用户的共性特征" | "请提取过去 6 个月流失用户的共性特征，回测准确率 > 70% 再实施" |
| 团队补缺 | "请先明确 {岗位} 的 {n} 个核心交付物再招聘" | "请先明确 CTO 的 3 个核心交付物再招聘" |
| 融资准备 | "请先准备 {n} 项关键材料再启动融资" | "请先准备财务预测 + 数据看板 + 客户案例 3 项关键材料再启动融资" |

```json
    }
  ],
  "summary": "当前阶段健康度 68%（亚健康），核心短板在获客渠道（CH）。建议优先验证内容营销渠道的 ROI，同时启动付费模式调整调研"
}
```

## 8. 认知层 5：验证认知

### 8.1 定位

验证认知层是认知链的**强制出口**，也是 M-BIZ 四层模型之外的 MealKey 独创扩展。该层负责：

1. 将 L4 输出的验证动作呈现给用户
2. 追踪验证进度
3. 接收验证结果并写入知识库
4. 基于验证结果触发认知链的重新推理

### 8.2 验证生命周期

```
[L4 输出验证动作]
    |
    v
[L5 创建验证任务]
  - 任务 ID
  - 关联的 strategic_suggestion
  - 验证截止日期
  - 状态: pending
    |
    v
[用户执行验证]  ← 用户在真实世界中执行
    |
    v
[用户回注结果]  ← 用户在 MealKey 中填写结果
  - 验证通过 / 部分通过 / 未通过
  - 实际数据
  - 经验总结
    |
    v
[L5 处理回注]
  - 更新验证任务状态
  - 将验证结果写入知识库（KAS）
  - 触发认知链重新推理（可选）
    |
    v
[认知升级]  ← 认知链的置信度提升 / 规则优化
```

### 8.3 验证任务结构

```json
{
  "verification_task_id": "VERIFY-001",
  "source_suggestion_id": "SUGGEST-001",
  "dimension": "CH",
  "verification_action": "请试运行 1 个月的小红书内容营销，统计 inbound leads 数量和 CPL",
  "estimated_period": "1 个月",
  "created_at": "2026-07-10T12:00:00Z",
  "deadline": "2026-08-10T12:00:00Z",
  "status": "pending",
  "reminder_schedule": ["7天后", "14天后", "截止前3天"]
}
```

### 8.4 回注数据结构

```json
{
  "verification_task_id": "VERIFY-001",
  "result": "partial_pass",
  "actual_data": {
    "leads_generated": 45,
    "cpl": 120,
    "conversion_rate": 0.08,
    "compared_to_previous_cac": 8000
  },
  "conclusion": "内容营销能带来流量，但转化率偏低，需要优化内容策略",
  "user_feedback": "执行过程发现目标客户在小红书上的决策链路比预期长",
  "new_insights": [
    "目标客户在购买前平均需要接触 5 次内容"
  ],
  "submitted_at": "2026-08-05T12:00:00Z"
}
```

### 8.5 回注后的认知升级

| 回注结果 | 认知链反应 |
|----------|-----------|
| 验证通过 | 提升相关规则的置信度，标记建议为"已验证有效" |
| 部分通过 | 优化验证动作参数，生成补充验证任务 |
| 未通过 | 降低相关规则权重，标记为"需复审"，触发 L2→L4 重新推理 |
| 用户提供新洞察 | 提取为新事实节点注入 L1，触发全链路增量推理 |

## 9. 认知链的链式推理引擎

### 9.1 推理引擎架构

```
创始人输入
    │
    ▼
┌──────────────────────────────────────────┐
│          认知链控制器 (ECC Controller)      │
│  负责：层间路由、状态管理、追问调度          │
└──────────────────────────────────────────┘
    │           │           │           │
    ▼           ▼           ▼           ▼
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ L1   │→│ L2   │→│ L3   │→│ L4   │→│ L5   │
│事实   │ │规则   │ │分析   │ │策略   │ │验证   │
└──────┘ └──────┘ └──────┘ └──────┘ └──────┘
    │           │           │           │
    ▼           ▼           ▼           ▼
┌──────────────────────────────────────────┐
│          知识资产体系 (KAS)                │
│  规则库 / 画像库 / 基准数据 / 验证记录    │
└──────────────────────────────────────────┘
```

### 9.2 推理策略

| 策略 | 触发场景 | 路径 |
|------|----------|------|
| 全链推理 | 首次评估、全面诊断 | L1→L2→L3→L4→L5 |
| 增量推理 | 用户补充新信息 | 相关认知层→后续层 |
| 定向推理 | 用户问特定问题 | 指定起始层→后续层 |
| 回溯推理 | 验证回注后 | 回注层→前置层重新验证→后续层更新 |
| 快速扫描 | 定时健康度检查 | L1(增量)→L2→L3(仅健康度) |

### 9.3 追问调度

当某层信息不足时，控制器暂停链路，发起追问：

```
L1: 事实不足 → 追问创始人补充
    ↓
L2: 规则冲突 → 追问确认细节
    ↓
L3: 模式匹配模糊 → 追问补充缺失维度
    ↓
L4: 策略优先级不确定 → 追问创始人偏好
    ↓
L5: 验证超时 → 自动提醒
```

## 10. 不算死账 — 验证认知作为强制出口

### 10.1 核心原则

**认知链不允许输出"死结论"。** 每一条策略层（L4）的输出都必须经过验证认知层（L5）的包裹，即附带可执行的验证动作。未经 L5 包裹的策略建议不进入用户视野。

### 10.2 合规检查流程

```
L4 生成策略建议
    │
    ▼
L5 合规检查器
    │
    ├── verification_action 存在？ → 否 → 拦截 + 触发重新生成
    │
    ├── 验证步骤可执行？ → 否 → 拦截 + 标记"需人工干预"
    │
    ├── 验证周期 ≤ 3 个月？ → 否 → 警告 + 建议拆分
    │
    ├── 结论可证伪？ → 否 → 拦截 + 要求重写
    │
    └── 全部通过 → 创建验证任务 → 呈现给用户
```

### 10.3 违规拦截的反馈

当认知链试图输出违规的"死建议"时，系统不对外暴露该建议，而是返回：

```json
{
  "status": "blocked",
  "reason": "L4 输出未通过 L5 '不算死账' 合规检查",
  "details": {
    "suggestion_id": "SUGGEST-003",
    "violation": "verification_action 不存在",
    "action": "建议包含 '建议优化营收结构' 属于空洞建议，请补充具体的验证动作"
  },
  "trigger": "重新生成策略建议"
}
```

### 10.4 验证完成率指标

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 验证完成率 | 已完成的验证任务 / 总验证任务 | ≥ 60% |
| 平均验证周期 | 从创建到完成的平均天数 | ≤ 45 天 |
| 回注率 | 完成验证后填写回注的比例 | ≥ 80% |
| 认知升级率 | 回注后触发认知链更新的比例 | ≥ 40% |

## 11. MealKey 场景适配

### 11.1 产品交互映射

| 认知层 | 产品形态 | 用户感知 |
|--------|----------|----------|

| L5 验证认知 | 验证任务看板 + 回注入口 | "我怎么验证这个判断对不对" |

### 11.2 创业阶段的感知与适配

MealKey FIL 自动感知创业阶段（Seed/Growth/Mature/Pivot），动态调整：

| 适配项 | 按阶段调整 |
|--------|-----------|
| 维度权重 | 见 §6.3 |
| 追问深度 | Seed 期追问更细，Pivot 期追问更快 |
| 验证周期 | Seed 期建议 2 周内可完成的验证 |
| 风险容忍度 | Seed 期降低风险预警阈值 |
| 建议语气 | Seed 期更引导性，Mature 期更直接 |

```
    ├── 认知报告
    │     ├── 健康度雷达图 (L3)
    │     ├── 规则判断明细 (L2)
    │     └── 策略行动清单 (L4 → L5)
    │
    └── 回注入口
          ├── 验证结果填写
          ├── 新洞察提交
          └── 认知升级对比
```
