## 0. 问题诊断：为什么现在像玩具

| 症状 | 根因 | 老板感知 |
|------|------|----------|
| 首页主推「示例店」 | 真数据路径弱于 Demo | 「玩玩而已」 |
| 导入靠粘贴 | 未接微信选文件 / 模板下载 | 门槛高、不专业 |
| 报告一次过 | 无历史档案、无 Δ 复检 | 用完即弃 |
| 评论靠手贴几条 | 无批量与可信度展示 | 体验官像凑数 |
| 学习回填偏藏 | DNA 变化不外显 | 「越用越懂」说了没看见 |
| 品类包只有合成账本 | 缺本店对标读数 | 像通用文案机 |
| Mealkey CTA 若过重 | 易被理解成「本体不完整」 | 已纠正为可选增强 |

**结论：** 引擎内核已专业（同源 `m-ops-diag`），玩具感来自**壳层数据闭环与经营节奏**，不是缺大模型。

## 1. 目标定义：什么叫「真真实实的能力」

小程序独立交付必须同时满足：

1. **用我的店、我的数**出完整会审（可拒签，不编故事）  
2. **第二次打开**能看见变化（Δ：变好/变坏/未变）  
3. **本周只做一件**可执行、可回填、可验证  
4. **证据链可见**（账本指标 #id + 评论主题计数）  
5. **门店 DNA 外显**（确认/否定主题权重）  
6. 全程**无 LLM、无付费墙、无 App 依赖**

Mealkey 只增强：档案云同步、解读追问、跨 Agent、决策回流——**不补课「能不能体检」**。

## 2. 能力地图（小程序必须自给）

```text
认识门店 → 采数（文件/粘贴/评论）→ 就绪度闸门
    → 规则体检（四官会审）→ 三层报告
    → 本周动作卡 → 学习回填 → 复检 Δ
    → 本机档案（多轮）→ （可选）导出分享
```

| 能力块 | 现状 | 目标态 | 优先级 |
|--------|------|--------|--------|
| A. 真店进件 | 浅字段 | 店名/城市/品类/人均/最想知道 + 焦点题 | P0 |
| B. 真数导入 | 仅粘贴 | 微信选 CSV/xlsx 文本 + 官方模板 + 校验报告 | P0 |
| C. 评论证据 | 粘贴打标 | 批量打标预览表 + 主题占比条 | P0 |
| D. 会审报告 | 有三层 | 证据下钻、拒签理由、就绪分置顶 | P0 |
| E. 复检 Δ | 无 | 存上次快照，强制对比页 | P0 |
| F. 学习 DNA | 有页偏弱 | 回填后权重条 + 「下一轮已加权」提示 | P0 |
| G. 本机档案 | 摘要级 | 历史列表（≥10 轮）可重开报告 | P1 |
| H. 动作卡分享 | 无 | 生成「本周一件事」图片/文案转发 | P1 |
| I. 品类阈值 | 提示文案 | 品类阈值表进引擎（晚市份额/TOP20 等） | P1 |
| J. 周提醒 | 无 | 订阅消息「该复检了」 | P2 |
| K. 外采 | 无 | 仅结构化粘贴/后续合规接口；禁止假外采 | P2 |

## 3. 分期执行方案

### 第一期（1–2 周）· 去玩具化底线 —— **必须做**

目标：老板用**自己的表**跑完，并愿意下周再来。

| ID | 事项 | 做法要点 | 验收 |
|----|------|----------|------|
| P0-1 | **文件导入** | `wx.chooseMessageFile` / 选文件读文本；保留粘贴兜底 | 不粘贴也能导入日明细 |
| P0-2 | **模板中心** | 日明细 / 菜销 / 菜单 三套 CSV 模板；一键复制表头 + 示例行说明 | 新用户 3 分钟内会填 |
| P0-3 | **导入质检单** | 行数、营业日、缺列、异常值 warnings 列表；硬门槛不达标则拒签说明 | 与四官拒签话术一致 |
| P0-4 | **示例降权** | 首页主 CTA =「体检我的店」；示例店收进「演示」折叠 | 主路径不再是 Demo |
| P0-5 | **复检 Δ** | 本地存 `lastSnapshot`（营收分解、主因、信号）；二次体检出对比模块 | 文案出现「较上次…」 |
| P0-6 | **DNA 外显** | 学习页/报告顶显示主题权重条；复扫后标「已按你的回填加权」 | 可见进化 |
| P0-7 | **评论预览** | 打标后展示 theme/sentiment 表，确认再跑诊 | 不盲跑 |

**第一期不做：** LLM、付费、假外采爬虫、原生复杂图表。

### 第二期（2–4 周）· 经营节奏 —— **像顾问工具**

| ID | 事项 | 验收 |
|----|------|------|
| P1-1 | 历史档案列表（时间、判定、主因、动作） | 可点开任一轮报告 |
| P1-2 | 「本周动作卡」一页：主因 + 唯一动作 + 验证问题；可复制/生成分享图 | 老板愿意发群/自己置顶 |
| P1-3 | 品类阈值进引擎（湘菜/火锅/茶饮）：高峰 MCI、TOP20%、客流降幅告警带 | 同数不同品类读数不同 |
| P1-4 | 就绪度仪表：硬门槛 / 强建议分项勾选进度 | 缺什么一眼知 |
| P1-5 | 浅检→深检升级路径：评论浅检后引导「补 7 天日明细解锁财务官」 | 漏斗清晰 |

### 第三期（并行/跨仓）· 与 Mealkey 的「更详尽」边界

小程序仍独立完整；下列**只在 Mealkey**做深，避免小程序变半成品：

- 云端多店、今日雷达、大模型解读、决策室  
- 合规外采 live 适配器（若做，结果回写小程序仅作「已同步证据包」）

小程序侧交付（不阻断报告）：

| ID | 事项 | 验收 |
|----|------|------|
| P2-1 | 能力对照页 `pages/enhance` + 交接证据包 JSON 导出 | 可复制 `mealkey.m-ops-diag.handoff.v1` |
| P2-2 | 复检提醒：本机 7 日到期 + 订阅消息占位（`recheckSubscribeTmplIds`） | 首页/报告可见到期提示 |
| P2-3 | 外采结构化粘贴（禁假爬虫） | 导入页合并进 evidence，明示不做抓取 |
| E-3 | 评论词典扩容（火锅/茶饮/外卖等） | 打标命中品类词 |

永不阻断报告；下载 App 始终为可选 CTA。

```text

  ├─ 体检我的店（主按钮）→ 进件 → 采数 → 出报告

  ├─ 独立体检已完成
  ├─ ①主因 ②本周一件事 ③较上次 Δ
  ├─ DNA 条
  ├─ 展开会审 / 学习回填
  ├─ 复检提醒 / 交接证据包
  └─ 可选 Mealkey（能力对照页）
```

## 6. 成功指标（4 周）

| 指标 | 玩具态 | 目标 |
|------|--------|------|
| 主路径使用真导入占比 | 低（示例为主） | ≥ 50% 完诊来自真导入或真评论 |
| 7 日复检率 | ≈0 | ≥ 20%（有档案的用户） |
| 学习回填率 | 低 | ≥ 30% 完诊用户点过成立/不成立 |
| 拒签理解 | 「坏了」 | 质检单后仍完成补数再诊 ≥ 40% |
| 「像玩具」主观反馈 | 高 | 内测老板口述「能对照我的账本」 |
