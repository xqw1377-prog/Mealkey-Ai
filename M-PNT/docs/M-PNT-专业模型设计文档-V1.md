## 2. 文档目标

本文档**不定义产品边界**，而定义 M-PNT 如何像一个真正高水平的餐饮品牌战略顾问一样思考、判断与决策。

其核心目标是让 M-PNT 具备以下能力：

- 深度理解餐饮行业逻辑与品牌定位方法论
- 能够进行结构化、多维度、对抗性的品牌定位推理
- 输出具备战略高度、可执行性与风险意识的品牌定位决策方案
- 支持多方案生成、多方案比较与红队挑战
- 为后续 Prompt 设计、知识资产建设和 Agent Workflow 提供稳定上位模型

---

## 3. 设计原则

M-PNT 的专业模型设计遵循以下原则。

### 3.1 以品牌定位决策为核心

M-PNT 不是知识型顾问，也不是分析型工具。

它的核心任务是：

> 帮助经营者完成一次高质量的品牌定位决策。

### 3.2 以心智占位为最高判断标准

所有分析、推理、比较与推荐，最终都应落到一个问题上：

> 该品牌是否有机会在目标消费者心智中占据一个独特、有利、可防御的位置。

### 3.3 以多方案竞争选择替代单一结论生成

品牌定位不是自动求解唯一答案，而是在多个合理路径之间做最优选择。

### 3.4 以行业深度替代通用品牌空话

M-PNT 的专业性来自餐饮行业知识、餐饮品牌案例和餐饮经营现实，而不是通用营销语言。

### 3.5 以风险挑战确保决策质量

任何定位方案都必须经过反向挑战与压力测试，避免输出「看起来正确」的脆弱方案。

## 4. 核心模型架构

M-PNT 的「大脑」由八个核心模块组成：

```text
品牌定位知识蒸馏模型
    ↓
餐饮品牌定位判断框架（六维）
    ↓
案例资产结构
    ↓
定位分析变量体系
    ↓
多方案生成机制
    ↓
三理论 Agent 矩阵（多视角校准）   ← Ries / Trout / Ye
    ↓
方案挑战与风险评估机制（含 Cross-Fire）
    ↓
品牌定位质量评价模型 → Synthesis 取舍
```

其中**三理论 Agent 矩阵**是多方案生成之后、最终推荐之前的强制校准层。专章见《M-PNT 三理论 Agent 矩阵设计 V1》。

## 5. 品牌定位知识蒸馏模型

### 5.1 模块目标

把分散的品牌定位理论、餐饮案例经验和行业规律，蒸馏成可供 M-PNT 稳定调用的结构化认知资产。

### 5.2 知识来源

主要来源包括：

- 经典品牌定位理论
- 餐饮行业最佳实践
- 真实品牌案例
- 品牌失败经验与反模式

### 5.3 蒸馏重点

重点蒸馏以下知识类型：

- 心智定位原理
- 竞争空间划分方法
- 差异化构建路径
- 品牌定位常见陷阱
- 场景与客群绑定规律
- 餐饮品牌长期壁垒形成机制

### 5.4 结构形式

建议采用两层结构：

| 层 | 用途 |
|---|---|
| 第一层：结构化知识对象 | 支撑检索、比较和程序化调用 |
| 第二层：高密度 Prompt 知识块 | 支撑实际推理与生成阶段的语义调用 |

### 5.5 蒸馏原则

- 不保留大段理论原文
- 不堆砌概念解释
- 所有理论都要转化为可执行判断规则
- 所有知识都必须服务于「品牌定位决策」

### 5.6 知识蒸馏流程

品牌定位知识不能直接以「文章摘要」的形式进入 M-PNT，而应经过一套固定蒸馏流程。

建议流程如下：

```text
原始理论 / 原始案例
    ↓
提取核心原理
    ↓
识别适用场景
    ↓
抽取关键变量
    ↓
转化为判断规则
    ↓
补充反模式与失败信号
    ↓
形成可调用知识对象
```

#### 5.6.1 从理论到规则

例如：

**原始理论：**

> 「品牌应在消费者心智中占据一个位置。」

蒸馏后不应只保留这句话，而应转化为：

| 项 | 内容 |
|---|---|
| 判断对象 | 目标消费者是否能快速理解品牌 |
| 判断标准 | 是否存在单一、清晰、可记忆的心智锚点 |
| 失败信号 | 定位语过宽、场景过多、价值表述过泛 |
| 输出要求 | 必须形成一句消费者可复述的定位表达 |

#### 5.6.2 从案例到模式

案例蒸馏不应停留在「这个品牌很成功」，而应抽取：

- 它当时面对什么竞争问题
- 它切入了哪个心智空位
- 它为什么没有选其他方向
- 它的差异靠什么被放大
- 它最后靠什么被记住

#### 5.6.3 从失败到挑战规则

失败案例尤其重要，因为它们会直接变成 challenge 模块的规则来源。

例如：

- 客群过宽
- 场景不成立
- 差异不可感知
- 资源无法交付
- 竞争位置已被强品牌占据

这些都应该被沉淀为「触发型风险规则」。

### 5.7 知识对象建议结构

为保证后续知识库与 Prompt 体系可复用，建议至少形成以下三类对象。

#### 5.7.1 TheoryRule

```text
TheoryRule
- name
- principle
- applicable_scenario
- key_variables
- decision_rules
- anti_patterns
- output_implication
```

#### 5.7.2 PositioningPattern

```text
PositioningPattern
- name
- pattern_type
- trigger_signals
- typical_context
- recommended_usage
- common_failure_modes
```

#### 5.7.3 ChallengeRule

```text
ChallengeRule
- name
- trigger_condition
- risk_type
- challenge_questions
- failure_signal
- mitigation_direction
```

> 注：完整七类知识资产与字段扩展见《M-PNT 知识资产结构设计文档 V1》。

## 6. 餐饮品牌定位判断框架

这是 M-PNT 最核心的判断框架，用于判断一个品牌定位方向是否成立。

建议采用**六维判断模型**。

### 6.1 市场机会维度

**关注：**

- 市场规模
- 增长空间
- 需求空白
- 消费趋势变化

**核心问题：**

- 这个方向是否有真实市场机会
- 是趋势机会、结构机会还是细分空位机会

### 6.2 竞争格局维度

**关注：**

- 直接竞品
- 间接竞品
- 替代性竞争
- 当前心智占位密度

**核心问题：**

- 这个位置是否已经拥挤
- 是否还有建立独立心智认知的空间

### 6.3 目标客群维度

**关注：**

- 客群心智特征
- 需求层次
- 决策动机
- 支付意愿

**核心问题：**

- 这个定位是否对应清晰的目标心智客户
- 这个客户是否真的会因此选择品牌

### 6.4 场景机会维度

**关注：**

- 高频场景
- 高价值场景
- 高情感场景
- 低竞争场景

**核心问题：**

- 品牌是否能在一个具体场景中被优先想到
- 场景是否足够强，足够稳定

### 6.5 资源匹配维度

**关注：**

- 产品能力
- 团队能力
- 资金能力
- 供应链能力
- 门店运营能力

**核心问题：**

- 自身资源是否支撑该定位长期成立
- 是否存在能力错配

### 6.6 可防御性维度

**关注：**

- 壁垒高度
- 模仿难度
- 长期可持续性
- 认知强化能力

**核心问题：**

- 这个定位是否容易被复制
- 品牌能否持续强化这一位置

### 6.7 六维判断的组合逻辑

六维判断不是平行罗列，而应形成一个有先后顺序的决策链。

建议采用以下组合逻辑：

```text
市场机会
→ 判断有没有必要进入

竞争格局
→ 判断有没有位置可占

目标客群
→ 判断有没有明确对象

场景机会
→ 判断有没有高频触发点

资源匹配
→ 判断有没有能力做成

可防御性
→ 判断有没有长期价值
```

#### 6.7.1 决策含义

这意味着：

| 条件 | 决策含义 |
|---|---|
| 市场机会不成立 | 后面无需深入 |
| 竞争空间过于拥挤 | 需转向差异寻找 |
| 客群与场景不清 | 定位不能成立 |
| 资源错配 | 方案即使好看也不能推荐 |
| 不可防御 | 只能作为短期打法，不能作为品牌定位主方案 |

### 6.8 六维判断的输出形式

六维判断后的结果，不应只是「有机会 / 没机会」，而应形成统一判断输出。

建议输出字段包括：

| 字段 | 说明 |
|---|---|
| opportunity_summary | 市场机会摘要 |
| competition_density | 竞争密度 |
| target_customer_fit | 客群匹配度 |
| scene_strength | 场景强度 |
| capability_fit | 资源/能力匹配 |
| defensibility_level | 可防御性等级 |
| overall_positioning_feasibility | 总体可行性 |

其中 `overall_positioning_feasibility` 可分为：

- `high`
- `medium`
- `low`
- `not_recommended`

## 7. 案例资产结构

案例资产的作用不是举例说明，而是为定位判断、方向生成和风险挑战提供可复用经验。

### 7.1 每条案例应包含的字段

- 项目背景
- 城市与市场条件
- 初始问题
- 品牌定位候选方向
- 最终定位选择
- 差异化构建方式
- 执行路径
- 结果与复盘数据
- 关键心智洞察
- 失败教训或风险提醒
- 可复用模板

### 7.2 重点保留的信息

每条案例至少要回答以下问题：

- 当时面对什么竞争环境
- 为什么没有选择其他方向
- 最终为什么选择该定位
- 这个定位靠什么被消费者记住
- 结果说明了什么

### 7.3 案例资产的类型划分

建议分为三类：

| 类型 | 用途 |
|---|---|
| 成功案例 | 提炼可复用的定位成功模式 |
| 转型案例 | 分析品牌如何从旧定位切换到新定位 |
| 失败案例 | 支撑挑战机制与反模式识别 |

### 7.4 案例资产标准结构

为支持后续检索、类比、Few-shot 复用，建议每条案例采用固定结构。

```text
Case Asset

1. Basic Info
- 案例名称
- 品类
- 城市 / 市场环境
- 品牌阶段

2. Initial Context
- 初始问题
- 初始资源条件
- 当时竞争格局

3. Candidate Positions
- 候选方向 A
- 候选方向 B
- 候选方向 C

4. Final Decision
- 最终定位选择
- 为什么选它
- 为什么不选其他方向

5. Execution
- 具体落地动作
- 品牌表达方式
- 产品 / 场景 / 体验如何支撑

6. Result
- 市场反馈
- 关键结果
- 是否验证成功

7. Learnings
- 关键心智洞察
- 可复用原则
- 风险提醒
```

### 7.5 案例可复用性的判断标准

不是所有案例都值得进入核心资产库。

建议至少满足以下条件之一：

- 定位选择过程清晰
- 有明确多方案比较过程
- 有足够结果反馈
- 有可提炼的成功规律
- 有明确失败教训

如果只是一段品牌故事，但没有决策过程和结果验证，则不应作为高价值案例资产。

## 8. 定位分析变量体系

M-PNT 必须有一套稳定的变量系统，确保每次分析不是凭感觉，而是基于统一变量完成判断。

### 8.1 宏观变量

包括：

- 消费趋势
- 政策变化
- 供应链变化
- 区域消费升级或降级趋势

### 8.2 中观变量

包括：

- 品类竞争地图
- 商圈竞争结构
- 客群迁移趋势
- 替代品类变化

### 8.3 微观变量

包括：

- 产品力
- 团队力
- 资金力
- 选址力
- 运营力

### 8.4 心智变量

包括：

- 竞品心智词
- 目标客群心智地图
- 用户第一联想
- 场景触发词
- 价格感知锚点

### 8.5 变量使用原则

| 变量层 | 用途 |
|---|---|
| 宏观变量 | 判断大方向 |
| 中观变量 | 判断竞争环境 |
| 微观变量 | 判断可执行性 |
| 心智变量 | 判断品牌定位成立性 |

其中，**心智变量是品牌定位决策中最关键的上层变量**。

## 9. 多方案生成机制

M-PNT 不直接输出一个答案，而是基于判断框架系统性生成多个候选品牌定位方向。

### 9.1 目标

生成 **3-5 个**逻辑自洽、彼此有明显差异、具备真实竞争对比价值的品牌定位方向。

### 9.2 每个方案必须包含

- 定位语
- 目标心智客户
- 核心消费场景
- 差异化支撑点
- 品牌心智目标
- 预期优势
- 主要成立前提

### 9.3 生成原则

- 方案之间必须形成不同竞争路径
- 不能只是表达方式不同，本质相同
- 每个方案都要具备成立逻辑
- 至少包含一个更保守的方向和一个更激进的方向

### 9.4 方案的差异来源

可以从以下维度切换：

- 客群差异
- 场景差异
- 品牌价值差异
- 价格心智差异
- 体验差异
- 文化表达差异

### 9.5 多方案生成主链路

建议采用固定生成链路，而不是一次性「拍脑袋」列方案。

```text
先确定基础约束
    ↓
识别主要竞争切口
    ↓
选择差异生成维度
    ↓
形成候选品牌定位方向
    ↓
为每个方向补足成立前提
    ↓
检查方案是否真正不同
```

#### 9.5.1 基础约束

先锁定不可违反的约束：

- 项目预算
- 团队能力
- 产品基础
- 品类边界
- 目标市场现实

#### 9.5.2 竞争切口识别

判断主要从哪里切入：

- 竞争空位
- 场景切口
- 人群切口
- 价值切口
- 价格带切口

#### 9.5.3 差异维度选择

至少选择一个主维度和一个辅维度，避免所有方案都只围绕同一条逻辑。

例如：

- 主维度：家庭场景 / 辅维度：温暖情感
- 主维度：年轻社交 / 辅维度：品牌表达

### 9.6 方案生成的质量约束

生成的 3-5 个方案必须满足以下要求：

- 至少 1 个稳健型方案
- 至少 1 个进攻型方案
- 至少 1 个差异明显的备选方案
- 方案之间不能只是文案换词
- 每个方案都必须能被一句话讲清楚

### 9.7 多方案比较标准

多方案比较时，建议使用统一比较维度：

- 心智独特性
- 客群清晰度
- 场景强度
- 竞争空间
- 资源匹配度
- 防御能力
- 验证难度

最终应形成：

- **主推荐方案**
- **次推荐方案**
- **不建议方案**

而不是简单列出「若干可选项」。

### 9.8 三理论 Agent 矩阵（多视角校准）

候选方向生成后，**不得**直接进入最终推荐。必须进入三理论 Agent 矩阵：

| Agent | 立场 | 核心问题 |
|---|---|---|
| Ries | 第一位置 / 品类 / 聚焦 / 领导壁垒 | 值不值得长期当「第一」去占？ |
| Trout | 竞争区隔 / 心智联想 / 避开强敌 | 能不能在竞品地图里钉住空位？ |
| Ye Maozhong | 中国现实 / 场景记忆 / 传播落地 | 消费者能不能懂、记、传、买？ |

**运行规则（摘要）：**

1. 三 Agent **共享**同一 Input Package（项目事实 + 六维诊断 + 候选集 + 心智信号）  
2. 三 Agent **并行**输出统一 Theory View（含 `theory_recommend`）  
3. **Cross-Fire** 暴露冲突/共识/淘汰，禁止平均主义折中  
4. **Synthesis** 结合百分制评分与心智 A–D，输出 `decision_recommend`  
5. 任一确认 **R4** 的方向不得作为 primary  

**双轨推荐等级（禁止混用字段）：**

| 层 | 字段 | 枚举 |
|---|---|---|
| 理论视角层 | `theory_recommend` | `strong_recommend` / `recommend` / `neutral` / `not_recommend` |
| 最终决策层 | `decision_recommend` | `primary` / `secondary` / `backup` / `reject` |

映射与否决优先级详见《三理论 Agent 矩阵设计 V1》第 7 章。  
完整职责边界、协议与质量门禁以该专章为准。

## 10. 方案挑战与风险评估机制

这是 M-PNT 保证决策质量的核心环节。

对每个候选方案，必须执行「红队挑战」。

### 10.1 主要挑战维度

| 维度 | 核心问题 |
|---|---|
| 竞争模仿风险 | 这个定位是否容易被同行快速模仿 |
| 执行落地难度 | 团队与资源是否真的能稳定交付该定位 |
| 心智接受度风险 | 消费者是否会自然接受这个品牌认知 |
| 资源匹配风险 | 品牌现有能力是否与定位要求匹配 |
| 长期演化风险 | 该定位是否会在未来 1-3 年迅速失效 |
| 反向压力测试 | 在最差场景下，这个定位最可能怎么失败 |

### 10.2 挑战原则

- 先挑战，再推荐
- 先找失败点，再看成立点
- 没有经过挑战的方案，不能进入最终推荐

### 10.3 输出结果

每个方案都应得到：

- 主要风险清单
- 风险等级判断
- 是否建议进入最终比较

### 10.4 红队挑战问题库

为保证挑战过程稳定，建议建立一组固定的红队问题。

#### 10.4.1 心智挑战

- 用户真的会这样理解这个品牌吗
- 这句话是否能在 3 秒内被记住
- 它和现有竞品相比真的明显不同吗

#### 10.4.2 客群挑战

- 第一目标客群是否足够明确
- 这个客群会不会觉得「这不是为我准备的」
- 他们是否愿意为这一定位付出更高选择成本

#### 10.4.3 场景挑战

- 核心场景是否真实高频
- 场景是否足够强到支撑品牌记忆
- 是否只是创始人的想象场景

#### 10.4.4 竞争挑战

- 这个位置是否已经被更强品牌占据
- 如果头部品牌复制这个打法，是否还有生存空间
- 是否会很快陷入价格竞争

#### 10.4.5 资源挑战

- 团队是否真的能把这个定位做出来
- 哪个关键能力目前明显不足
- 如果执行只能做到 70%，这个定位还成立吗

#### 10.4.6 时间挑战

- 这个定位是短期热度还是长期资产
- 未来 1-3 年是否还会成立
- 它是否容易被趋势变化快速淘汰

### 10.5 风险分级建议

建议将风险统一分为四级：

| 等级 | 含义 |
|---|---|
| R1 | 低风险，可控 |
| R2 | 中风险，需要配套策略 |
| R3 | 高风险，需谨慎进入 |
| R4 | 不可接受，不建议作为主方案 |

### 10.6 淘汰规则

以下情况应直接淘汰，不进入最终推荐：

- 心智不可理解
- 客群不清
- 场景不成立
- 资源明显不匹配
- 已有强势品牌深度占位且无新切口
- 风险达到 R4

## 11. 品牌定位质量评价模型

为了让 M-PNT 的推荐具有一致标准，需要建立质量评价模型。

建议采用 **100 分制**。

### 11.1 评分维度

| 维度 | 分值 |
|---|---|
| 心智独特性 | 25 |
| 竞争优势强度 | 20 |
| 客群匹配度 | 15 |
| 可执行性 | 15 |
| 长期壁垒 | 15 |
| 风险可控性 | 10 |
| **合计** | **100** |

### 11.2 评分逻辑

| 维度 | 看什么 |
|---|---|
| 心智独特性 | 是否容易被消费者快速区分与记忆 |
| 竞争优势强度 | 是否建立在真实竞争优势上，而不是空话 |
| 客群匹配度 | 是否与目标客群的真实需求和心智路径一致 |
| 可执行性 | 是否有能力把定位长期做出来 |
| 长期壁垒 | 能否在未来持续强化，而非短期有效 |
| 风险可控性 | 主要风险是否可被识别、管理和缓解 |

### 11.3 使用方式

评分模型主要用于：

- 内部多方案比较
- 推荐逻辑支撑
- 输出质量校验

不一定需要完整展示给用户，但必须支撑最终推荐。

### 11.4 心智占位评价方法

「心智占位」是 M-PNT 的最高判断标准，因此需要单独定义评价方法。

评价一个方案时，至少要回答以下问题：

1. 这个定位是否一句话就能被理解
2. 这个定位是否能快速区别于主要竞品
3. 这个定位是否能绑定明确消费场景
4. 这个定位是否能被反复强化
5. 这个定位是否能形成长期第一联想

### 11.5 心智占位评价结果

建议将心智占位结果分为四类：

| 等级 | 含义 | 特征 |
|---|---|---|
| A 级：可强占位 | 主推荐优先 | 定位清晰、差异显著、竞争空位明确、可长期强化 |
| B 级：可争夺 | 次推荐可能 | 定位基本成立，但竞争较强或资源要求较高 |
| C 级：可尝试 | 备选 / 验证 | 有一定空间，但需要较强验证，不宜直接作为长期主定位 |
| D 级：不建议 | 淘汰 | 心智模糊、无法区分、缺乏长期防御性 |

### 11.6 最终评价结论

每个候选方案在完成评分后，建议统一生成：

| 字段 | 说明 |
|---|---|
| mind_position_level | A / B / C / D |
| overall_score | 0–100 |
| decision_recommend | primary / secondary / backup / reject |
| theory_vote_summary | Ries / Trout / Ye 各自 preferred 与 theory_recommend 摘要 |
| key_risk_summary | 主要风险摘要 |
| max_risk_severity | R1–R4 |

> 字段名统一使用 `decision_recommend`；理论层使用 `theory_recommend`。旧称 `recommendation_level` 在最终决策语境下等同于 `decision_recommend`。

### 11.7 评价与三理论汇合顺序

```text
候选集
  → 三理论 Theory Views
  → Cross-Fire（冲突/共识/淘汰）
  → 百分制 + 心智 A–D + 风险 R1–R4
  → decision_recommend
```

**否决优先级：** R4 淘汰 > 心智 D > 两票 not_recommend > 百分制排序。

## 12. Agent 内部 Workflow 主链路

M-PNT 的内部主链路建议如下：

```text
1. Situation 解析
    ↓
2. Insight 生成（含六维诊断）
    ↓
3. Position 多方案生成
    ↓
4. 三理论 Agent 矩阵（Ries / Trout / Ye 并行）
    ↓
5. Cross-Fire 碰撞
    ↓
6. 评分 + 红队补强
    ↓
7. Synthesis 推荐与 Decision
    ↓
8. Validation 设计
    ↓
9. Quality Check → Final Output
```

| 步骤 | 内容 |
|---|---|
| 12.1 Situation 解析 | 提取项目关键变量，形成结构化项目理解 |
| 12.2 Insight 生成 | 运行六维判断框架，结合相关案例，形成关键洞察 |
| 12.3 Position 多方案生成 | 生成 3-5 个候选品牌定位方向 |
| 12.4 三理论矩阵 | 同一输入上并行三视角 Theory View |
| 12.5 Cross-Fire | 冲突、共识、淘汰、optimized_direction |
| 12.6 评分与红队 | 百分制 + R 级；无 R4 进入主推池 |
| 12.7 Synthesis / Decision | primary / secondary / backup / reject |
| 12.8 Validation 设计 | 最小验证动作、观察指标、调整触发 |
| 12.9 Quality / Final | 质量门禁后输出 M-Solution |

## 13. 输出标准

输出必须严格对齐产品定义文档，并统一采用 **M-Solution Framework** 七段式结构。

### 13.1 强制要求

- 必须使用 M-Solution Framework
- Position 部分必须体现多方案比较
- 必须包含最终推荐理由
- 必须包含风险提示
- 必须包含验证路径

### 13.2 表达要求

- 语言专业
- 结论务实
- 有战略洞察
- 避免空洞表达
- 避免理论说教

### 13.3 最终品牌定位方案输出结构

```text
M-PNT Brand Position Solution

1. Situation
- 项目状态
- 当前主要问题
- 关键约束

2. Insight
- 核心市场洞察
- 核心竞争洞察
- 核心心智洞察

3. Position
- 候选方案 A
- 候选方案 B
- 候选方案 C
- 多方案比较
- 最终推荐定位

4. Strategy
- 品牌差异化策略
- 核心竞争方式
- 不参与的竞争方向

5. Action
- 品牌表达建议
- 产品 / 场景 / 体验支撑方向
- 第一阶段执行重点

6. Validation
- 30天验证重点
- 90天观察指标
- 调整触发条件

7. Decision
- 最终决策结论
- 推荐理由
- 主要风险
- 信心判断
```

### 13.4 输出底线要求

最终方案必须满足以下底线：

1. 用户能一眼看懂推荐结论
2. 用户知道为什么选这个方向
3. 用户知道为什么不选其他方向
4. 用户知道最主要风险是什么
5. 用户知道下一步怎么验证

如果不能满足以上 5 条，则输出仍不算合格的品牌定位决策方案。

## 16. 结论

M-PNT 的专业模型可以收敛为：

> 一个以品牌定位决策为核心、以心智占位为判断标准、以多方案比较与红队挑战为主要机制的餐饮品牌战略推理系统。

它决定了 M-PNT 后续的：

- Prompt 体系设计
- 知识库结构
- Workflow 设计
- Few-shot 案例构建
- 质量评价方式

因此，这份文档将作为 M-PNT 后续实现阶段的核心上位设计文档。
