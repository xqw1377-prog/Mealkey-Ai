# M-PNT 交互原型框架 V0.1

> 基于现有引擎能力、三理论矩阵、`MKDecision` 输出结构与上一轮交互形态构想，收敛出的第一版可执行原型框架。

## 1. 原型目标

这份原型框架不是视觉稿，也不是前端实现稿。

它的目标不是把 M-PNT 做成独立产品，而是把它嵌回 MealKey 母体里，成为餐饮项目决策链中的专项能力。

更准确地说，这一版要实现的是：

> 把 M-PNT 从“会输出结果的定位引擎”，变成 **MealKey 内部用于产出可落地定位策略与执行方案的专项模块**。

因此，这一版只定义四件事：

1. MealKey 内的挂载位置
2. 状态机
3. 核心模块关系
4. 桌面端 / 移动端低保真布局

---

## 2. 产品定位一句话

M-PNT 不是独立前台产品，也不是单独使用的定位工具，它在 MealKey 里的定位应该是：

> **MealKey 经营会议中的“餐饮定位策略与执行方案”专项模块。**

---

## 3. 原型边界

## 3.1 这一版要解决什么

- 用户第一次进入时，知道自己身处哪一步
- 用户能快速看懂当前主推定位
- 用户能比较候选方向，而不是只看一段长文
- 用户能理解三理论为什么分歧、为什么最终不是折中
- 用户不只拿到定位结论，还能拿到可执行方案
- 用户能持续看到 `Next Step`
- 用户能在 30/90 天后回来复审，而不是重新开题

## 3.2 这一版先不解决什么

- 高保真视觉规范
- 复杂动画
- 实时协作
- 多项目跨空间管理
- 完整设计系统

---

## 4. MealKey 内的挂载结构

不再定义“独立 M-PNT 产品路由”，而是定义 M-PNT 在 MealKey 里的 4 个挂载位置。

## 4.1 挂载清单

### A. Dashboard / Today 定位状态卡

作用：

- 展示当前项目是否已有定位结论
- 展示复审提醒
- 进入本轮经营会议

### B. Advisor / 经营会议主界面中的定位模块

作用：

- 在经营会议里调起 M-PNT
- 查看候选方向、三理论交锋、最终裁决
- 输出定位策略与执行方案

这是整个交互的核心挂载位。

### C. Project Profile / 项目档案中的定位卡与历史版本

作用：

- 查看当前定位结论
- 查看定位策略与执行方案
- 查看历史版本与 diff

### D. Decisions / Review Queue 中的复审入口

作用：

- 展示待复审的定位任务
- 基于验证结果重新发起定位复审会议

## 5. 状态机

M-PNT 的交互不应该靠“独立页面切换感”驱动，而应该靠“MealKey 项目状态 + 定位决策状态”驱动。

## 5.1 状态定义

### `idle`

还没有形成定位任务。

页面特征：

- Dashboard / Advisor 中还没有定位结论
- 主动作是：`发起定位分析`

### `fact_collecting`

正在补齐项目事实。

页面特征：

- 重点不是长表单，而是从项目上下文中补足最少必要事实
- 系统会标记缺失项

### `running`

M-PNT 正在执行 7 步工作流。

页面特征：

- 以会议内步骤进度呈现
- 当前只展示必要中间状态
- 不开放大面积自由编辑

### `decision_ready`

本轮定位结论已经形成。

页面特征：

- 主推定位固定显示
- 候选方向、理论交锋、最终裁决完整展开
- 定位策略与执行方案完整展开
- `Next Step` 常驻

### `validating`

定位已进入验证期。

页面特征：

- 强调 30 天 / 90 天验证动作
- 强调风险和 kill criteria

### `review_due`

到了复审节点。

页面特征：

- 显示需要复审的原因
- 支持一键发起复审会议

### `reviewing`

正在基于旧结论重新审视。

页面特征：

- 默认展示旧版结论
- 强调这轮变化点

## 6. MealKey 内骨架总览

## 6.1 Advisor / 经营会议桌面端布局

采用 `顶部状态条 + 三栏工作区`。

```text
+----------------------------------------------------------------------------------+
| MealKey 会议顶部锚定区                                                            |
| 主推定位 | decision_recommend | 风险等级 | 当前 Next Step | 发起复审 | 继续讨论   |
+----------------------+-------------------------------------------+----------------+
| 左侧项目上下文面板    | 中部定位策略区                              | 右侧执行行动栏    |
| 项目摘要              | 候选方向卡组                                | 今日唯一问题     |
| 品类/城市/商圈        | 三理论交锋                                  | 本周执行动作     |
| 预算/阶段             | 最终裁决                                    | 30天验证         |
| 创始人优势/盲区       | 定位策略                                    | kill criteria    |
| 缺失事实提醒          | 执行方案                                    | 复审入口         |
|                       | 结论版本记录                                | 基于定位继续讨论 |
+----------------------+-------------------------------------------+----------------+
```

### 布局比例建议

- 左栏：`280px`
- 中栏：`minmax(760px, 1fr)`
- 右栏：`320px`

---

## 6.2 MealKey 移动端布局

移动端不能照搬三栏，采用：

> `顶部固定条 + 单列主阅读区 + 底部固定 Next Step`

```text
+----------------------------------+
| 顶部固定决策条                    |
| 主推定位 / 风险 / 复审           |
+----------------------------------+
| 主阅读区                         |
| 1. 当前结论                      |
| 2. 候选方向                      |
| 3. 三理论交锋                    |
| 4. 定位策略                      |
| 5. 执行方案                      |
| 6. 验证动作                      |
+----------------------------------+
| 底部固定行动条                    |
| 今日下一步 | 发起复审 | 继续讨论  |
+----------------------------------+
```

### 移动端约束

- 主阅读区高度至少保留 `40vh`
- 所有次要信息默认折叠
- 不允许底部堆满按钮墙
- `Next Step` 必须始终可触达

## 7. 核心模块定义

## 7.1 模块 A：顶部固定决策条

这是 MealKey 会议里的锚定条，不是独立 M-PNT 导航。

### 字段

- 当前主推定位
- `decision_recommend`
- `max_risk_severity`
- 当前阶段状态
- 当前 `Next Step`

### 动作

- `发起复审`
- `基于该定位继续讨论`

### 交互原则

- 不随滚动消失
- 始终只显示 1 个主推定位
- 风险必须和结论并列，而不是藏到下方

---

## 7.2 模块 B：项目事实面板

用于承载项目上下文，不要求用户脱离 MealKey 重新填一遍。

### 显示内容

- 品类 / 城市 / 商圈
- 预算 / 项目阶段
- 创始人经验 / 优势 / 盲区
- 当前假设场景
- 缺失事实提醒

### 交互动作

- 展开完整事实
- 标记事实不足
- 进入补充信息

### 状态

- `complete`
- `partial`
- `critical_missing`

---

## 7.3 模块 C：候选方向卡组

这是定位比较区，不是结果列表。

### 每张卡必须包含

- 方案名
- 一句话定位
- 类型：稳健型 / 进攻型 / 备选型
- 切口类型
- 主优势
- 主风险
- 当前建议等级：`primary / secondary / backup / reject`

### 卡片操作

- `查看理由`
- `查看理论票`
- `加入对比`

### 排序规则

1. `primary`
2. `secondary`
3. `backup`
4. `reject`

---

## 7.4 模块 D：三理论交锋区

这是 M-PNT 的核心差异化模块，也是它在 MealKey 里最该被看见的部分。

### 结构

三列并排：

- Ries
- Trout
- Ye

每列只保留：

- 偏好方向
- 关键理由
- 主要反对点
- 风险等级

下方单独有 `Cross-Fire 摘要`：

- 主要冲突
- 共识点
- 淘汰项
- 为什么最终不是折中

### 交互动作

- 切换查看某一候选在三理论下的票型
- 查看某个理论的完整论证
- 高亮 `R4`

### 交互原则

- 默认先看结论，再允许展开完整辩论
- 冲突优先展示，不做教学式大段理论解释

---

## 7.5 模块 E：最终裁决区

这是页面中心。

### 必须固定展示 4 块内容

1. `选这个`
2. `为什么选`
3. `为什么不选别的`
4. `接下来怎么验证`

### 字段映射建议

- `summary / mentalPosition` → 选这个
- `why_choose_this` → 为什么选
- `why_not_others` → 为什么不选别的
- `validation + nextSteps` → 接下来怎么验证

---

## 7.6 模块 F：定位策略区

这里解决的是“定在哪里”。

### 必须包含

- 核心定位一句话
- 目标心智客户
- 核心消费场景
- 竞争位置
- 差异化钉子

### 字段映射建议

- `brandPositioning.category`
- `brandPositioning.targetCustomers`
- `brandPositioning.differentiation`
- `brandPositioning.mentalPosition`
- `summary`

---

## 7.7 模块 G：执行方案区

这里解决的是“接下来怎么落地”。

### 必须包含

- 品牌表达策略
- 菜单 / 产品锚点
- 空间 / 场景承接
- 首阶段传播动作
- 30 / 90 天执行里程碑

### 字段映射建议

- `mSolution.strategy`
- `mSolution.action`
- `nextSteps`
- `validation`

---

## 7.8 模块 H：验证与复审区

这是 M-PNT 从一次性报告走向长期价值的关键。

### 内容结构

- 30 天验证
- 90 天验证
- kill criteria
- 上一版 / 当前版差异
- 复审入口

### 交互动作

- 标记某项验证已完成
- 发起复审
- 查看版本 diff

## 8. 关键用户流

## 8.1 首次定位用户流

```text
进入 MealKey Dashboard / Advisor
  → 发起定位分析
  → 从项目上下文补齐最少必要事实
  → 运行 M-PNT
  → 在经营会议中查看候选方向
  → 查看三理论交锋
  → 形成最终裁决
  → 输出定位策略与执行方案
  → 锁定 30 天验证动作
```

## 8.2 老项目复审用户流

```text
进入 Dashboard / Decisions / Project Profile
  → 看到上次定位结论
  → 看到复审提醒
  → 进入复审会议
  → 查看新旧定位 diff
  → 发起复审会议
  → 产出新版本决策
```

## 11. 交互原则

### 原则 1：永远把“当前推荐 + 怎么落地”放在第一屏

用户不是来学习流程的，是来拿走可执行的定位策略和执行动作的。

### 原则 2：冲突比过程更重要

不是把所有推理都搬上来，而是优先展示：

- 谁支持
- 谁反对
- 反对是否构成否决

### 原则 3：下一步必须常驻

`Next Step` 不能被滚动淹没。

### 原则 4：默认收敛，按需展开

长文本、完整论证、历史版本都应折叠在第二层。

### 原则 5：移动端优先保住主阅读区

不能让按钮、输入区、底栏挤压掉主结论区域。

---

## 12. 原型实现优先级

如果下一步直接进入前端实现，建议顺序如下：

1. 先做 Advisor 会议里的定位模块
2. 再做 Dashboard / Today 定位状态卡
3. 再做 Project Profile / Decisions 的复审挂载
4. 最后补首次定位事实补全态

原因很简单：

Advisor 会议里的定位模块，是 M-PNT 价值最集中的地方。

## 13. 这份框架的交付结果

到这里，M-PNT 的交互原型已经从“方向判断”推进到了“可画、可做、可对接”的骨架层。

下一轮如果继续推进，最自然的动作就是：

1. 按这份框架出低保真页面
2. 直接做静态前端原型
3. 再把 `MKDecision` 数据映射接进去
