# 登录失败调试记录

## 症状
- 部署后访问登录页可以打开。
- 输入账号密码后提示“登录失败，请稍后重试”。
- 需要确认失败发生在前端提交、认证回调、服务端鉴权还是部署环境配置。

## 当前假设
1. 线上缺少认证相关环境变量，导致登录接口在服务端直接失败。
2. `NEXTAUTH_URL`、回调域名或 Cookie 域配置错误，导致登录后会话未能建立。
3. 部署环境数据库连接异常，导致查用户或写 session 时报错。
4. 认证提供方的 `authorize`/登录 API 返回了通用错误，被前端统一映射成“请稍后重试”。
5. 线上构建使用了错误的 base URL、secret 或 edge/node 运行时配置，导致登录请求在 Vercel 上行为与本地不一致。

## 计划
- 先复现线上登录并抓取网络/控制台证据。
- 再回到代码审查认证配置、登录路由和环境变量依赖。
- 如需进一步确认，再最小化加埋点并给出修复。

## 已收集证据
- 登录页前端只在 `signIn("credentials")` 抛异常时展示“登录失败，请稍后重试。”。
- 若是普通登录失败（如密码错误），前端展示的是“邮箱或密码不正确，请重新输入。”。
- 线上 Vercel 地址 `https://traeucfpws48.vercel.app/login` 复测后，页面留在登录页并展示“登录失败，请稍后重试。”。
- 对认证回调等价复核后拿到服务端响应：`429`，响应体为 `{"error":"登录尝试过于频繁，请稍后再试"}`。
- 认证路由 `POST /api/auth/[...nextauth]` 在进入 Auth.js 处理前，先按 IP 做 `30 次 / 15 分钟` 的限流。
- `authorize()` 内部还会再按邮箱做 `10 次 / 15 分钟` 的限流。

## 当前判断
- 当前直接阻塞登录的是限流，不是前端表单提交失败。
- 前端错误提示存在误导：`429` 被 `catch` 吞成了“登录失败，请稍后重试。”，用户看不到“登录尝试过于频繁”的真实原因。
- 还需要进一步确认是：
  - 该 IP 的真实尝试次数确实过多；
  - 还是生产环境里取不到真实 IP，导致多个请求都落到 `unknown` 或共享键上，被异常集中限流。

## 新证据（2026-07-21）
- 修复限流 fail-closed 后，登录页错误已从“登录失败，请稍后重试”变为“邮箱或密码不正确，请重新输入”，说明系统级 429 闸门已解除，进入真实账号校验分支。
- 正式域名 `http://mealkey.cn/` 浏览器与命令行复测均为 `502`，且无跳转。
- 正式域名 `https://mealkey.cn/` 浏览器报 `ERR_CONNECTION_CLOSED`，连接在拿到有效响应前即被关闭。
- 本地 `.vercel/project.json` 绑定项目为 `trae_ucfpws48`，`orgId = team_UO60HCdRTvLCNiDgYzfXqx0E`。
- 但当前 CLI 执行 `vercel domains inspect mealkey.cn` 返回：`Domain not found by "mealkey.cn" under xqw1377-progs-projects`。

## 更新判断
- 登录代码层的核心阻塞已缓解；剩余两层问题已分离：
  1. 正式域名入口异常（`mealkey.cn` 不是正常指向当前可控项目，或 CDN / 证书 / 回源配置坏了）。
  2. 当前测试账号凭证本身未通过生产库校验。
- 现阶段要“恢复可登录”，优先顺序应为：
  1. 修复 `mealkey.cn` 域名绑定与 HTTPS 可达性；
  2. 再处理账号密码恢复或找回密码链路。
