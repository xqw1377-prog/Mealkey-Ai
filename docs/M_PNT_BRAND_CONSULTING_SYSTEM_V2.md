# M-PNT Brand Consulting System V2

> 模块：M-PNT  
> 定位：品牌战略咨询中心  
> 核心判断：M-PNT 的问题不是能力不足，而是咨询体验层没有被 Runtime 统一起来

## 一、核心定位重新定义

旧：

> AI 品牌定位 Agent

问题：

用户感知仍然是：

> 问 AI -> 得答案

价值太弱。

新：

> **AI 品牌战略咨询机构**

一句话：

> 帮助餐饮创业者完成从市场认知、品牌定位到落地路径设计的一次完整品牌战略咨询。

用户不是在使用工具。

而是在：

> 进入一个虚拟咨询团队。

## 二、M-PNT 咨询团队架构

未来的 M-PNT 不是一个 Agent，而是一家 AI 原生品牌咨询机构。

```text
                M-PNT Brand Consulting

                       Chief Brand Officer
                              |
              --------------------------------
              |              |               |
       Research Director   Strategy Board   Implementation
              |              |               |
        市场研究顾问      定位专家委员会       落地顾问
```

### 1. Brand Research Agent

角色：

品牌研究总监。

负责：

> 理解企业和市场

输入不是一次填表，而是顾问访谈。

例如：

AI：

> 我先了解一下你的品牌，你为什么想做这个项目？

用户：

> 我觉得现在年轻人喜欢湘菜。

AI 继续追问：

> 你的竞争对手是谁？  
> 为什么消费者选择你，而不是他们？

最终形成：

## Brand Brief

```text
品牌名称：
品类：
城市：
经营规模：
目标消费者：
当前问题：
竞争对手：
创业初心：
商业目标：
```

## 三、咨询第一阶段：品牌诊断

状态：

```text
PROJECT_INIT

↓

BRAND_DISCOVERY
```

用户看到的不是“请输入资料”，而是：

```text
你好，我将作为你的品牌顾问。

在开始定位前，
我需要先理解你的品牌。

预计需要 10 分钟。

完成后我会生成：
《品牌战略诊断报告》
```

用户感知是：

> 我已经进入咨询流程

## 四、第二阶段：市场诊断

### Market Intelligence Agent

职责不是搜索资料，而是回答：

> 这个市场给不给你机会

输入：

`Brand Brief`

分析六个维度：

```text
1. 品类趋势
2. 消费需求变化
3. 竞争格局
4. 用户心智空位
5. 价格机会
6. 场景机会
```

输出：

## 《市场机会诊断》

例如：

```text
当前判断：

湘菜市场仍有机会

但传统湘菜竞争高度集中

机会不在“更正宗”

而在：

年轻聚餐场景
```

这里是用户第一次真正看到 AI 的价值。

## 五、第三阶段：三大定位专家委员会

这是 M-PNT 的核心资产。

保留三位理论顾问：

### 里斯

代表：

心智占位理论。

关注：

- 第一
- 唯一
- 品类代表

### 特劳特

代表：

竞争定位理论。

关注：

- 对手是谁
- 差异在哪里
- 如何攻击

### 叶茂中

代表：

营销洞察理论。

关注：

- 消费者情绪
- 场景
- 记忆点

重点不是三个聊天机器人，而是：

## 专家独立提案会议

流程：

```text
市场资料

↓

三专家分别分析

↓

各自产生定位方案

↓

进入战略会议
```

每个专家都输出：

- 定位观点
- 核心判断
- 依据
- 优势
- 风险
- 反对其他方案理由

例如：

里斯：

```text
建议：
年轻湘菜第一品牌

理由：
占据年轻湘菜心智

风险：
竞争者容易复制
```

特劳特：

```text
建议：
区别于传统湘菜的聚会湘菜

理由：
避开红海竞争

风险：
品类认知弱
```

叶茂中：

```text
建议：
年轻人的热辣社交场

理由：
情绪驱动消费

风险：
长期资产不足
```

## 六、第四阶段：专家战略会议

这里是现在 `Cross-Fire` 要升级的地方。

不是“争论表演”，而是：

## 模拟咨询委员会

流程：

```text
Proposal A

↓

Proposal B

↓

Proposal C

↓

冲突分析

↓

战略取舍

↓

形成共识
```

输出：

## 定位委员会决议

例如：

```text
委员会认为：

三个方案共同指向：
年轻湘菜消费升级机会。


最终建议：

定位：
年轻人的热辣聚餐品牌


原因：

兼顾：

心智占位
+
竞争差异
+
消费动力
```

## 七、第五阶段：用户参与确认

```text

现在需要你的判断。

A
B
C

为什么？
```

用户必须能够：

- 选择
- 修改
- 补充

最终形成：

## Human + AI 共识

## 十二、总体判断

现在 M-PNT 最大的问题不是：

> 三个专家不够强

而是：

**缺少一次真实咨询体验。**

你现在已经把“专家大脑”做出来了。

下一阶段真正要做的是：

> 把专家大脑组织成一家咨询公司

这样用户感受到的就不再是：

> 在和 AI 聊天

而会变成：

> 正在接受一次品牌战略咨询

## 六、Theory Agent 统一接口

当前三位专家不能只是三个 Prompt。

应该统一为：

## TheoryAgent

输入：

- `BrandContext`
- `MarketEvidence`
- `UserGoal`

输出：

## TheoryDecision

```typescript
type TheoryDecision = {
  theory: "Ries" | "Trout" | "YeMaozhong";
  coreInsight: string;
  positioningOption: string;
  evidence: string[];
  confidence: number;
  challenge: string;
};
```

这意味着：

- 每个专家都必须给出核心洞察
- 每个专家都必须提出自己的定位方向
- 每个专家都必须带证据
- 每个专家都必须指出对其他理论的挑战

## 七、Cross-Fire Engine 重构

当前的 Cross-Fire 不能再是大师语录式 `if-else`。

应该重构成真正的：

## Cross-Fire Engine

输入：

- `Ries Proposal`
- `Trout Proposal`
- `Ye Proposal`

输出：

## Conflict Matrix

维度包括：

1. 战略冲突  
2. 理论优势  
3. 理论风险  
4. 适用条件

例如：

```text
里斯：
应该成为年轻湘菜第一品牌

特劳特：
不能只讲年轻，需要攻击竞争位置

叶茂中：
需要创造消费场景
```

系统输出：

```text
共识：

年轻湘菜不是最终定位

最终定位：

年轻人聚餐场景中的湘菜代表


原因：

满足品类占位
+
竞争差异
+
情绪连接
```

### Cross-Fire 目标

不要模拟专家语气。

而要生成：

- 冲突结构
- 共识逻辑
- 裁决理由

## 八、状态机设计

页面不能驱动流程，状态机才是咨询 Runtime 的事实源。

### 建议状态机

```typescript
enum ConsultingProjectState {
  INIT,
  PROFILE_COLLECTING,
  MARKET_DIAGNOSIS,
  EXPERT_ANALYSIS,
  DEBATE,
  CONSENSUS,
  USER_CONFIRM,
  IMPLEMENTATION
}
```

### 状态含义

- `INIT`：项目启动
- `PROFILE_COLLECTING`：品牌访谈与建档
- `MARKET_DIAGNOSIS`：市场与品类诊断
- `EXPERT_ANALYSIS`：专家独立判断
- `DEBATE`：理论交叉辩论
- `CONSENSUS`：形成委员会建议
- `USER_CONFIRM`：老板确认战略方向
- `IMPLEMENTATION`：生成落地路径

## 九、Decision 节点

必须明确哪些地方用户必须参与。

### 必须参与的 3 个节点

#### 1. 诊断反馈

系统问：

> 以上诊断是否符合你的真实情况？

用户输入会影响：

- 品牌诊断修正
- 访谈补问

#### 2. 方案选择

系统问：

> 三个方向中，哪个更符合你的长期愿景？

用户输入会影响：

- 最终定位方向
- 创始人偏好

#### 3. 战略确认

系统问：

> 你是否确认把这个方向作为当前品牌战略？

用户输入会影响：

- 是否写入正式战略资产
- 是否生成落地路径

## 十、Runtime 事件设计

M-PNT V2 应该有明确事件，而不是只产出页面文案。

### 核心事件

```typescript
type M_PntRuntimeEvent =
  | "BrandProfileCreated"
  | "BrandDiagnosisCompleted"
  | "MarketAnalysisCompleted"
  | "TheoryProposalGenerated"
  | "DebateStarted"
  | "ConsensusGenerated"
  | "FounderDecisionRecorded"
  | "StrategyConfirmed"
  | "ImplementationPlanned";
```

### 事件作用

- 驱动状态变化
- 驱动 UI 投影
- 写入 Memory
- 为后续平台观测提供事实源

## 十二、为什么 market-intel 迁移不是当前最高优先级

`market-intel` 未来一定会独立到：

```text
knowledge/
|
├── market-assets
├── category-assets
├── consumer-assets
├── case-assets
```

方向是对的。

但当前更高优先级不是拆数据，而是：

> 先让一次咨询流程跑通

也就是说，M-PNT 当前最先要统一的是咨询 Workflow，不是知识目录。

## 十三、为什么 distillation 不应该立刻删

`distillation` 可以暂缓删除。

因为它未来可能成为：

## Knowledge Distillation Layer

也就是：

```text
行业知识

↓

理论模型

↓

判断规则

↓

案例经验

↓

Agent 能力
```

这层以后会决定：

> AI 如何把行业知识变成判断能力

所以它更像待重构资产，而不是应被简单删除的废弃模块。

## 十四、最终判断

M-PNT 当前真正缺的不是更多 AI 能力，而是：

## 咨询流程操作系统

也就是：

- 用户如何进入咨询项目
- 顾问团队如何分阶段出现
- 决策节点如何被用户感知
- 战略资产如何被持续积累

换句话说：

**M-PNT 后端已经达到“引擎级”，下一阶段不是继续增强 AI，而是把它包装成一次真实咨询服务。**
